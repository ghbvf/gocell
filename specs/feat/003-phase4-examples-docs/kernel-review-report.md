### D. Kernel Integration Health -- YELLOW

This is the most critical finding of this review. The Phase 4 spec and kernel-constraints.md constraint C-19 prescribed "kernel/ zero code modification (or only doc/comments)." However, the actual diff shows **364 lines changed across 20 kernel files**, including substantive logic changes:

**kernel/cell/base.go** (+41 lines, -3 lines):
- Added `sync.Mutex` to BaseCell for concurrent safety on Init/Start/Stop/Health/Ready
- Added `shutdownCtx`/`shutdownCancel` context pair for graceful shutdown signaling
- New public method `ShutdownCtx()` -- this is an API surface expansion
- All state-accessing methods (Init, Start, Stop, Health, Ready) now hold the mutex

**kernel/assembly/assembly.go** (+4 lines, -3 lines):
- Stop state guard changed from `state == stateStopping` to `state != stateStarted`
- This is a behavior change: previously Stop was only blocked during a concurrent Stop; now Stop is a no-op from any non-Started state

**kernel/governance/rules_fmt.go** (+55 lines):
- New `validateFMT10()` rule: banned legacy field names check + slash-separator detection in contract IDs
- New `bannedFieldNames` map

**kernel/governance/validate.go** (+1 line):
- Added `v.validateFMT10()` call in the validation chain

**kernel/metadata/parser.go** (+20 lines):
- Empty-ID guards added for cell, slice, contract, journey, assembly parsing

**kernel/outbox/outbox.go** (+6 lines):
- Doc comment expansion on Entry.ID and Writer.Write (doc-only, no signature change)

**doc.go files** (10 files, +33 lines total):
- Package-level godoc additions -- doc-only

**Test files** (assembly_test.go +23 lines, base_test.go +49 lines, governance/validate_test.go +53 lines, metadata/parser_test.go +76 lines):
- All new tests covering the added functionality

**Assessment of the changes**:

The kernel modifications are technically sound and beneficial:
1. The mutex on BaseCell addresses a real concurrency safety gap

3. FMT-10 catches legacy field names in metadata
4. Empty-ID guards prevent silent corruption in parsing

However, these changes violate the "kernel zero modification" constraint that was explicitly established in the kernel-constraints report for Phase 4. The changes were not flagged as constraint violations during S6 review -- the QA report (AC-16.1) states "kernel files unchanged from Phase 3" which is factually incorrect based on the git diff.

- `kernel/outbox/outbox.go`: only doc comment changes on Writer.Write. Signature unchanged.
- New method `ShutdownCtx()` is on BaseCell (concrete type), not on the Cell interface. No interface break.

## Must-Fix Items for Post-v1.0 (3 items max)

Phase 4 is the final Phase before v1.0 release. The following items cannot be fixed within Phase 4 scope but must be addressed post-release.

### 1. [P4-TD-05] Outbox full-chain integration test (post-v1.0)

**Constraint**: C-14
**Impact**: The L2 consistency promise -- business write + outbox write in same transaction, relay to message broker, consumer with idempotency -- has never been validated end-to-end in a single test. Individual adapter tests prove each leg works in isolation, but the full chain (postgres outbox write -> relay poll -> rabbitmq publish -> consumer subscribe -> redis idempotency check) is untested. This is the most critical verification gap for the L2 architecture.
**Action**: Create `TestIntegration_OutboxFullChain` spanning all 3 adapters in a single test using testcontainers multi-container composition.

### 2. [P4-TD-04] order-cell L2 consistency declaration without enforcement (post-v1.0)

**Constraint**: Related to C-12
**Impact**: order-cell declares `consistencyLevel: L2` in both cell.yaml and code but does not enforce outboxWriter presence in Init (unlike access-core, audit-core, config-core which all fail-fast). The order-create service uses `publisher.Publish()` directly instead of transactional outbox writes. This means the example that is supposed to be the "golden path" for building a custom Cell demonstrates a weaker consistency pattern than the framework's built-in cells. Framework evaluators following the tutorial would learn an incorrect pattern.
**Action**: Either enforce outboxWriter in order-cell Init (matching built-in cell behavior) or downgrade to L1 and document the rationale for when to choose L1 vs L2.

### 3. [P4-TD-06] CI example validation `|| true` silently swallows errors (post-v1.0)

**Constraint**: Related to C-22
**Impact**: The CI workflow's example metadata validation step uses `|| true`, meaning any validation error in example cell.yaml/slice.yaml/contract.yaml files is silently ignored. This undermines the metadata compliance guarantee -- the main project metadata is strictly validated, but example project metadata validation is cosmetic. As examples grow and evolve, metadata drift would go undetected.
**Action**: Remove `|| true` from the example validation loop. If certain examples intentionally deviate from strict validation, use explicit waiver annotations rather than blanket error suppression.
