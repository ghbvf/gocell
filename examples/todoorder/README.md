# todoorder Example

A minimal order management application demonstrating the GoCell "golden path":
creating a business Cell with HTTP endpoints and a switchable event emission path.

## Architecture

- **ordercell** (L2 OutboxFact): manages order lifecycle
  - **ordercreate** slice: POST creates an order and emits the order creation event
  - **orderquery** slice: GET retrieves orders by ID or lists all

## Runtime Modes

| Capability | Demo mode in this example | Durable mode when fully wired |
|---|---|---|
| Order storage | In-memory repository | Durable repository |
| HTTP API | Enabled | Enabled |
| Event emission path | Direct in-process publish via `eventbus.New()` | Transactional outbox write + relay publish |
| Event durability | No | Yes |
| Event identity source | Generated by the in-memory bus | `outbox.Entry.ID` |
| Recommended use | Local exploration and tests | Real L2 deployments |

This example application runs in demo mode. The reusable `ordercell` code can also emit a real `outbox.Entry` when `WithOutboxWriter(...)` and `WithTxManager(...)` are injected.

## Quick Start (In-Memory Mode)

No external dependencies required. Uses an in-memory repository and in-process event bus.

```bash
go run ./examples/todoorder
```

The server starts with primary listener on `:8082` (API + infra) and internal listener on `:9082` (control-plane).

## Docker Mode

Start infrastructure services, then run the application:

```bash
cd examples/todoorder
docker compose up -d
cd ../..
go run ./examples/todoorder
```

Docker mode here only starts surrounding infrastructure. The example still runs in demo mode unless the application is changed to inject a real outbox writer, transaction manager, and relay-backed publisher.

## API

### Authentication (demo mode)

PR-CFG-C made `RoleCustomer` mandatory for every `/api/v1/orders/*` route, so
the example installs a tiny `demoTokenVerifier` in `main.go` that accepts a
single hard-coded bearer token. Export it once and reuse it in every curl
invocation below:

```bash
export TODOORDER_TOKEN=todoorder-customer-demo-token
```

Anonymous calls (no `Authorization: Bearer ...` header) receive `401 Unauthorized`;
calls with a different token receive `401`; valid tokens missing `role:customer`
would receive `403 Forbidden`. The demo verifier always issues the customer
role — to exercise the 403 path, replace the verifier or strip
`ordercell.RoleCustomer` from the issued claims.

### Create an order

```bash
curl -X POST http://localhost:8082/api/v1/orders/ \
  -H "Authorization: Bearer $TODOORDER_TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"item":"test"}'
```

Response (201):

```json
{"data":{"id":"ord-...","item":"test","status":"pending"}}
```

### List all orders

```bash
curl -H "Authorization: Bearer $TODOORDER_TOKEN" \
  http://localhost:8082/api/v1/orders/
```

Response (200):

```json
{"data":[...],"nextCursor":"","hasMore":false}
```

When another page exists, `nextCursor` contains the opaque token for the next request.

### Get order by ID

```bash
curl -H "Authorization: Bearer $TODOORDER_TOKEN" \
  http://localhost:8082/api/v1/orders/{id}
```

Response (200):

```json
{"data":{"id":"ord-...","item":"test","status":"pending","createdAt":"..."}}
```

## Durable Wiring Checklist

To move from demo mode to a durable L2 path, wire all of the following into the example application:

1. A durable repository instead of the in-memory repository.
2. `ordercell.WithOutboxWriter(...)`.
3. `ordercell.WithTxManager(...)`.
4. A relay-backed broker publisher/subscriber pair instead of the in-process event bus.

### Health check

```bash
curl http://localhost:8082/healthz
curl http://localhost:8082/readyz
curl -H "X-Readyz-Token: $GOCELL_READYZ_VERBOSE_TOKEN" 'http://localhost:8082/readyz?verbose'
```

`/healthz` is liveness-only. Use `/readyz?verbose` for the detailed readiness breakdown — PR-A35 requires `GOCELL_READYZ_VERBOSE_TOKEN` to be set and the matching `X-Readyz-Token` header on the request.
