## §1. 筛选原则

> 不是「主流框架有就要补」。3 条筛选线：

1. **example 已暴露真实信号** 或 **v1 corebundle 平台承诺暗含**（不补就是平台缺口）
2. **零信任 / MDM 后端服务建立的必要前提**（框架层基础不补，业务 cell 建不起来）
3. **沿用 SPI-REGISTRY 范式**（同族 ≥3 / examples ≥5 / 跨 binary 部署 任一达成才起 PR）

**框架层 vs 业务层切分**（关键）：
- ✅ **框架层应补**：kernel / runtime / adapters / cmd/gocell / tools 内的能力
- ❌ **业务层不补**：放 `cells/{zerotrust,mdm}*` 的业务 cell，不属于 GoCell 框架职责

按上述切分，下面列出**框架应补的 15 项**（v1.1 P1 5 项 + v1.2 P2 6 项 + 零信任 ZT 5 项 + MDM 框架增量 4 项中扣除重叠 = 15 项），业务 cell 列在 §6 仅作参考。

## §4. 零信任专属框架增量（ZT 系列 / 5 PR）

> 触发条件：v1.1 ship 后；零信任后端服务建设启动。

参考标准：**NIST SP 800-207 ZTA** + **BeyondCorp** + **CISA ZTMM**。

| # | PR | 缺它就建不起来什么 | 方案 | 工时 | 依赖 |
|---|---|---|---|---|---|
| ZT-1 | **PR-ZT-MTLS-TRANSPORT** | 设备 mTLS 认证；服务身份 | runtime/transport 加证书装载（CA pool / cert reload / SAN 校验）；与 V11-2 gRPC 集成 | 24h + 12h | V11-2 |
| ZT-2 | **PR-ZT-PKI-PROTOCOL-LAYER** | 设备证书自动签发（MDM enrollment 必需）| `adapters/pki` 实现 SCEP / EST / ACME 协议层；下游接 vault PKI backend | 40h + 16h | ZT-1 |
| ZT-3 | **PR-ZT-OPA-PDP-INTEGRATION** | 外置政策中心（ABAC + Rego）；compliance 评估 | runtime/auth 扩展 PDP hook；新建 `adapters/opa` HTTP/embedded 双模式；现有 `runtime/auth.AuthPlan` 与 OPA Decision 互通 | 24h + 12h | 独立 |
| ZT-4 | **PR-ZT-SPIFFE-WORKLOAD-ID** | 跨集群 service-to-service 身份；零信任服务网格场景 | `adapters/spiffe` 实现 Workload API；与 ZT-1 mTLS listener 互通 | 32h + 12h | ZT-1 |
| ZT-5 | **PR-ZT-SIEM-EXPORTER** | 合规审计外送 SOC（CEF / LEEF / syslog）| `runtime/observability/siem` 新增 exporter；auditcore 输出可订阅 | 16h + 8h | 独立 |

**累计**：~136h dev + 60h review ≈ 196h / **6-8 周双 lane**

**不做的零信任组件**（与定位张力）：
- ❌ 微分段 / 网络策略 — 网络层而非应用层；走 K8s NetworkPolicy / Cilium
- ❌ DNS / Proxy 层（BeyondCorp 风格）— 部署架构层；用 Pomerium / Teleport / OpenZiti

> 触发条件：ZT 系列基础就绪后；MDM 后端服务建设启动。

参考标准：**Apple MDM Protocol** + **Android Management API** + **Windows MDM (OMA-DM)** + **AppConfig**。

| # | PR | 缺它就建不起来什么 | 方案 | 工时 | 依赖 |
|---|---|---|---|---|---|
| MDM-1 | **PR-MDM-PUSH-CHANNEL** | MDM control channel（Apple MDM 用 APNs 唤醒；Android 用 FCM；Windows 用 WNS）| `adapters/push` 三家统一接口；HTTP/2 长连接 + token 管理；幂等通过 outbox | 32h + 12h | 独立 |
| MDM-2 | **PR-MDM-LDAP-SCIM** | 企业用户 / 组同步源（AD / LDAP / SCIM）| `adapters/ldap` + accesscore 加 `scimprovider` slice（实现 SCIM 2.0 provider）| 40h + 16h | V11-5 tenant |
| MDM-3 | **PR-MDM-DEVICE-ATTESTATION** | TPM / Keystore / SecureEnclave 签名验证（设备 posture 来源）| `runtime/crypto` 扩展 TPM 2.0 quote / Android Key Attestation / Apple DeviceCheck 验证 helper | 24h + 12h | ZT-1 mTLS |
| MDM-4 | **PR-MDM-S3-SIGNED-URL-CDN** | 大文件分发（IPA / APK / MSI 多 GB）| `adapters/s3` 增强 signed URL TTL + range 下载支持 + CDN origin helper（CloudFront / 七牛 / 阿里云 OSS 兼容）| 8h + 4h | 独立 |

**不做的 MDM 组件**（业务 / 边界外）：
- ❌ 远程协助 / screen share — 业务功能，不在框架范畴
- ❌ VPN 配置 / Per-app VPN — 操作系统配置层；MDM 后端只下发 profile，不实现 VPN

## §6. 业务 cell（**不是框架范畴**，仅参考）

放 `cells/{zerotrust,mdm}*` 或独立仓库的业务实现，**不算 GoCell 框架补充能力**：

**零信任业务 cell**（3 项）：
- `cells/devicecore` — 设备身份注册 + 生命周期（依赖 ZT-1, ZT-2, MDM-3）
- `cells/riskscore` — 风险评分引擎（依赖 ZT-3 OPA + 设备 posture + 行为信号）
- `cells/policycontrol` — PDP 政策托管 cell（依赖 ZT-3 OPA）

**MDM 业务 cell**（5 项）：
- `cells/enrollmentcore` — Apple DEP/ABM、Android EMM、Windows Autopilot enrollment（依赖 ZT-2 SCEP, V11-3 saga）
- `cells/inventorycore` — 设备清单 + 状态机（依赖 V11-3 saga）
- `cells/profilecore` — `.mobileconfig` / Android JSON / Windows CSP profile orchestrator（依赖 inventorycore）
- `cells/compliancecore` — compliance evaluator（依赖 ZT-3 OPA）
- `cells/appcatalog` — app 分发 (MAM)（依赖 MDM-4）

**判断逻辑**：能用 contract.yaml + slice.yaml + handler.go + service.go 写出来的 = 业务 cell；要改 kernel/ 或 runtime/ 或新建 adapters/ 的 = 框架增量。

## §7. 能力域级永久 won't-do

| 能力 | 不做理由 |
|---|---|
| Service Mesh（Istio sidecar 等） | 与 Cell 模型重叠；framework 内做 sidecar 是反向定位 |
| 管理控制台 / Admin UI | J1 unified catalog API 已部分覆盖；UI 由消费者自建（gocell-web sibling repo 已是范式）|
| 国际化 i18n | 不在框架定位；业务 cell 自决（go-i18n 直接引用即可）|
| GraphQL 服务端 | 无明确触发；REST + contract 已能聚合；ssobff 用 BFF 模式更贴 cell 边界 |
| DI 容器（reflect-based） | 与 029 won't-do E8 同类；assembly 显式 wire 是定位选择，reflect 注入破坏 codegen 链 |
| 远程注册中心 adapter | 触发条件 = ≥2 独立 binary 部署 + 跨 binary 服务发现需求；当前 1 corebundle 模型未达 |
| 多 cloud KMS（AWS/GCP/Azure 全家桶） | 与 029 won't-do S14a 同类；部署目标云平台明确后单家触发，不预先全做 |
| A/B 实验平台 | featureflag 已 cover 80% 场景；experiment 设计是产品层而非框架层 |
| WebSocket presence/room/广播抽象 | adapter 低层够用；业务 cell 自实现（centrifuge 直接引用即可）|
| 混沌工程注入 | toxiproxy / chaos-mesh 现成工具；framework 内做 chaos injector 是反向定位 |
| 微分段 / 网络策略 | 网络层而非应用层；K8s NetworkPolicy / Cilium 已是标准 |
| BeyondCorp DNS / Proxy 层 | 部署架构层；Pomerium / Teleport / OpenZiti 现成方案 |
| 远程协助 / VPN 配置 | MDM 后端边界外（OS 配置层）|

## §8. 实施次序与依赖图

```
v1.0 ship (029 K Phase 4 末)
  │
  ▼
v1.1 P1 框架核心基础 (§2, 5 PR / 6-8 周)
  ├─ V11-1 MFA ────────────┐
  ├─ V11-2 gRPC ──┬────────┤
  ├─ V11-3 Saga ──┘ (依赖 V11-2)
  ├─ V11-4 MQTT (独立)     │
  └─ V11-5 Tenant (独立)   │
  │                         │
  ▼                         │
v1.2 P2 触发条件待达 (§3, 按需)│
  └─ V12-1..V12-6 (随客户)  │
                            │
─── 零信任 / MDM 框架基础线 ───
                            ▼
ZT 系列 (§4, 5 PR / 6-8 周)
  ├─ ZT-1 mTLS ────┬───────┐
  ├─ ZT-2 PKI ─────┘       │
  ├─ ZT-3 OPA PDP (独立)   │
  ├─ ZT-4 SPIFFE (依赖 ZT-1)
  └─ ZT-5 SIEM (独立)      │
                            │
                            ▼
MDM 框架增量 (§5, 4 PR / 4-5 周)
  ├─ MDM-1 Push (独立)
  ├─ MDM-2 LDAP/SCIM (依赖 V11-5)
  ├─ MDM-3 Attestation (依赖 ZT-1)
  └─ MDM-4 S3 CDN (独立)
                            │
                            ▼
零信任 / MDM 业务 cell (§6, 不在框架范畴)
  └─ devicecore / riskscore / policycontrol
  └─ enrollmentcore / inventorycore / profilecore / compliancecore / appcatalog
```

**关键约束**：

1. **ZT 必须先于 MDM**：ZT-2 PKI 是 MDM-1 enrollment 前提；ZT-3 OPA 是 MDM compliance 前提；ZT-1 mTLS 是 MDM-3 attestation 前提
2. **V11-3 Saga 是 MDM 灵魂**：MDM 命令下发→执行→ack 是天然 saga；Saga 没成熟 MDM 状态机会脆弱
3. **V11-5 Tenant 不能后补**：MDM 典型多企业 SaaS，tenant 隔离是基础，加在中后期会触发 cell 全面重写

**累计周期估算**：
- v1.1 P1：~6-8 周（双 lane）
- ZT 系列：~6-8 周（双 lane）
- MDM 框架增量：~4-5 周（双 lane）
- **框架层总计**：~16-21 周（约 4-5 个月双 lane 并行 + reviewer buffer）
- 业务 cell 层：另算（不是框架职责）
