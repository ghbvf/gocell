## §1. 问题陈述

GoCell 当前 100% 是「monorepo 内部消费」形态：
- ✅ `cells/{accesscore,auditcore,configcore}` 是仓库内 cell
- ✅ `examples/{demo,iotdevice,ssobff,todoorder}` 是仓库内 example
- ❌ **没有任何「外部 repo import gocell + 自写 cell」的真实路径验证**

当外部项目（如零信任 / MDM 后端服务）要消费 GoCell 时，会立即遇到：

| 问题 | 现状 | 影响 |
|---|---|---|
| **目录结构不知道怎么摆** | 无 guide；examples/ 是 monorepo 形态非外部消费形态 | 新项目 1-2 天试错；社区 PR 可能错位 |
| **kernel/runtime/adapters 边界不清** | 没说外部消费者要不要复刻这些目录 | 误以为要建 `kernel/` `runtime/` 平行目录 |
| **adapters 调用链不清** | 没文档说明 cell 不能 import adapter；composition root 概念缺位 | 易写出违反依赖反转的代码 |
| **自建 adapter 放哪里没规则** | 顶层 `adapters/` vs `cells/{cell}/internal/adapters/` 选择无指导 | 复用粒度判断错误 |
| **公开 API surface 不明** | gocell 大量包都可被 import，没标注哪些 stable / experimental | 外部消费者 import 后框架 break 风险高 |
| **脚手架弱** | `gocell scaffold cell --id=...` 只造单 cell；不造完整 bundle / cell 库 | 30 分钟跑通承诺难兑现 |
| **archtest 不可复用** | `tools/archtest/` 是 internal 测试，不是公开包 | 外部项目无法继承 layer 治理规则 |
| **没有外部消费 e2e 验证** | CI 全在 monorepo 内跑 | 接口变更对外部消费方影响盲区 |

## §2. 三种外部消费形态（参考）

经讨论确定三种形态。详细目录树与代码示例见本文 §6。

| 形态 | 用途 | 输出 | 必有目录 | 禁有目录 |
|---|---|---|---|---|
| **A · 单 cell 库** | 写一个 cell 给团队 / 第三方复用 | 可 import 的 Go 包 | `cells/{id}/` | `cmd/` `assemblies/` `kernel/` `runtime/` |
| **B · 完整应用 / Bundle** | 生产部署（零信任 / MDM 后端服务）| 可执行二进制 | `cmd/` `cells/` `contracts/` `journeys/` `assemblies/` `actors.yaml` | `kernel/` `runtime/` |
| **C · 可复用 cell 集合** | 中间层库（如 `gocell-zerotrust`）| Go module（多 cell + contracts）| `cells/` `contracts/` | `cmd/` `assemblies/` `kernel/` `runtime/` |

**核心规则（archtest 守护）**：
1. 消费者**不复刻 `kernel/` `runtime/`**（来自 gocell import；命名是 GoCell 内部 layering，不是通用约定）
2. **`cells/{cell}/` 不 import 任何 `adapters/`**（包括 gocell 的和自建的）
3. **`cmd/{bundle}/` 是唯一同时 import `cells/` + `adapters/` 的地方**（composition root）
4. 跨 cell 私有共享代码用 `internal/`；公开扩展点用 `pkg/`（Go 标准约定，非 GoCell 命名）

### DX-2 · 脚手架扩展三种形态（在 K#09 之上扩）

**问题**：029 K#09 PR-V1-SCAFFOLD-ONE-CMD 只规划了 `gocell new-cell` / `new-slice` / `new-contract` 单组件粒度，缺「整个 bundle」「整个 cell 库」级别的脚手架。

**方案**：在 K#09 之上扩展两个子命令：

```bash
# 形态 A：单 cell 库
gocell new-celllib <name> --type=core --level=L1 --module=github.com/myorg/<name>
# → 生成 go.mod + cells/{name}/ 完整骨架 + README

# 形态 B：完整应用 / Bundle
gocell new-bundle <name> --module=github.com/myorg/<name> --include-cells=devicecore,policycontrol
# → 生成 cmd/{name}/ + cells/ + contracts/ + journeys/ + assemblies/ + Makefile + docker-compose.yml

# 形态 C：cell 集合（中间层）
gocell new-cellset <name> --module=github.com/myorg/gocell-<group>
# → 生成 cells/ 多 cell 骨架 + contracts/ + adapters/（可选）+ consumer-guide.md 模板
```

**工时**：16h + 8h
**ship**：L2
**依赖**：029 K#09 ship 后
**触发**：v1.0 ship 后

### DX-3 · 公开 API surface 标记 + 稳定性承诺

**问题**：gocell 仓库 102 个生产包都可被外部 import，没有「哪些 stable / 哪些 experimental / 哪些 internal-only」的承诺。外部消费方 import 后任何 commit 都可能 break。

**方案**：
- 在每个包 `doc.go` 加 stability tag：
  ```go
  // Package outbox provides the framework-level reliable event publishing.
  //
  // API stability: stable (v1+, semver-compatible)
  // External consumers may import this package directly.
  package outbox
  ```
- 三档稳定性：`stable` / `experimental` / `internal`
- archtest 加 gate：标记 `internal` 的包不能被 `cells/` `examples/` `cmd/` 之外 import（防止误把内部包暴露）
- `gocell graph --filter=public` 只输出 stable + experimental 包
- 文档：`docs/guides/api-stability.md` 列全 stable 包清单（≈30-40 个包）

**工时**：12h + 6h
**ship**：L2
**依赖**：无
**触发**：v1.0 ship 后立即（API 稳定承诺是开源项目硬要求）

---

### DX-4 · archtest 公开化（触发条件待达）

**问题**：`tools/archtest/` 当前是 internal 测试，外部项目无法 import 同样的 layer 治理规则。但 archtest 是 GoCell 核心差异化能力之一（158 条规则），不让外部用是浪费。

**方案**：
- 把 layer 治理规则（LAYER / REF / TOPO / VERIFY 部分）提取到 `pkg/archtest/`（新 public 包）
- 暴露 `archtest.NewLayerChecker(modulePath, gocellPath)` API
- 外部项目 `tools/archtest/layer_test.go`：
  ```go
  import "github.com/ghbvf/gocell/pkg/archtest"

  func TestLayerCompliance(t *testing.T) {
      checker := archtest.NewLayerChecker("github.com/myorg/zerotrust-platform")
      checker.AssertNoCellImportsAdapter(t)
      checker.AssertCmdIsCompositionRoot(t)
  }
  ```
- 与 DX-1 guide 联动：guide 示例代码直接用此 API

**工时**：24h + 12h
**ship**：L3
**依赖**：DX-3（公开包 surface 锁定）
**触发条件（任一达成）**：(a) 第一个外部消费 repo 出现；(b) DX-1 文档里出现「外部 repo 怎么跑 archtest」反复提问

### DX-5 · 外部消费 e2e 验证（CI 守护）

**问题**：当前 CI 全在 monorepo 内跑；GoCell 公开 API break 时无法第一时间感知。

**方案**：
- CI workflow 新增 `external-consumer-smoke.yaml`：
  ```yaml
  jobs:
    external-consumer:
      runs-on: ubuntu-latest
      steps:
        - run: go install github.com/ghbvf/gocell/cmd/gocell@${{ github.sha }}
        - run: |
            mkdir /tmp/external-test && cd /tmp/external-test
            gocell new-bundle test-bundle --module=example.com/test
            cd test-bundle
            go mod tidy
            go build ./...
            go test ./...
            ./scripts/verify-archtest.sh   # 复用 DX-4 archtest
  ```
- 跑三种形态各一次（A/B/C）
- 失败时 PR block

**工时**：16h + 8h
**ship**：L2
**依赖**：DX-2（new-bundle 命令）+ DX-4（archtest 可外部用）
**触发**：DX-2 + DX-4 ship 后

---

### DX-6 · cookiecutter 模板（complement DX-2）

**问题**：CLI 脚手架（DX-2）适合简单场景；复杂 bundle（含 PG/Redis/Vault wire）需要更完整模板。

**方案**：
- 新建 `templates/external/`
  - `templates/external/celllib/` 形态 A 模板
  - `templates/external/bundle/` 形态 B 模板（含 cmd/ + 完整 corebundle 接线 + docker-compose）
  - `templates/external/cellset/` 形态 C 模板
- `gocell new-bundle --from-template=full` 用 cookiecutter 模板而非内置 stub
- 模板内含完整 README + Makefile + .env.example + .github/workflows/

**工时**：12h + 6h
**ship**：L2
**依赖**：DX-2 框架
**触发**：与 DX-2 同窗口

## §4. 实施次序

```
v1.0 ship (029 K Phase 4 末)
   ↓
立即启动（DX-1 / DX-3 独立可起）：
   ├─ DX-1 外部消费者 guide       (8h + 4h, L1)
   └─ DX-3 公开 API surface 标记  (12h + 6h, L2)
   ↓
v1.1 P1 ship 后 + 029 K#09 ship 后：
   ├─ DX-2 脚手架扩展            (16h + 8h, L2, 依赖 K#09)
   └─ DX-6 cookiecutter 模板     (12h + 6h, L2, 与 DX-2 同窗口)
   ↓
触发条件达成后：
   ├─ DX-4 archtest 公开化       (24h + 12h, L3, 触发 = 第一个外部 repo)
   └─ DX-5 外部消费 e2e          (16h + 8h, L2, 依赖 DX-2 + DX-4)
```

**关键判断**：
1. **DX-1 + DX-3 是 v1.0 ship 后的「头两件事」**（单线 1 周可全 ship）；它们决定外部消费者第一接触体验
2. **DX-2 + DX-6 一定要绑 029 K#09**（K#09 已规划单组件 scaffold，扩成三形态成本最低）
3. **DX-4 + DX-5 是「有外部消费者后再做」**（无人外部消费时投入这两项 ROI 低）

## §6. 三种形态完整参考（详）

> 本节是规范化的目录树参考，DX-1 PR 可直接 fork 此节内容到 `docs/guides/external-consumer-layouts.md`。

### 6.1 形态 A · 单 cell 库

```
my-cell/                                     module: github.com/myorg/my-cell
├── go.mod                                   require github.com/ghbvf/gocell vX.Y.Z
├── go.sum
├── README.md
└── cells/
    └── mycell/                              ★ cell ID = 包名（小写，Go 合法标识符）
        ├── cell.yaml
        ├── cell.go
        ├── cell_test.go
        ├── cell_init.go
        ├── cell_routes.go
        ├── options.go
        ├── internal/
        │   ├── domain/
        │   ├── ports/
        │   ├── dto/
        │   └── mem/
        └── slices/
            └── myaction/
                ├── slice.yaml
                ├── handler.go
                ├── service.go
                └── service_test.go
```

**禁止**：`cmd/` `main.go` `assemblies/` `kernel/` `runtime/`、顶层 `adapters/`

### 6.2 形态 B · 完整应用 / Bundle（生产部署）

```
my-zerotrust-platform/                       module: github.com/myorg/zerotrust-platform
├── go.mod                                   require github.com/ghbvf/gocell vX.Y.Z
├── go.sum
├── README.md / Makefile / docker-compose.yml / .env.example
│
├── cmd/                                     ★ composition root 唯一位置
│   └── platform/
│       ├── main.go                          唯一 import gocell/adapters/* 的位置
│       ├── shared_deps.go
│       └── *_module.go                      per-cell wire
│
├── cells/                                   ★ 一级平铺，cell ID 全局唯一
│   ├── devicecore/
│   │   ├── cell.yaml / cell.go
│   │   ├── postgres/                        cell 自带 PG repo（如有，与 internal 同级）
│   │   ├── internal/
│   │   │   ├── domain/ ports/ dto/
│   │   │   └── adapters/                    ⚠ 仅 cell 专属 adapter（不复用）
│   │   └── slices/
│   ├── policycontrol/
│   └── compliancecore/
│
├── contracts/                               ★ kind 在外，domain 在内
│   ├── http/ event/ command/ projection/ shared/
│
├── journeys/                                ★ 平铺，文件名前缀区分
│   ├── J-*.yaml
│   └── status-board.yaml
│
├── assemblies/
│   └── platform/                            与 cmd/platform/ 同名
│       ├── assembly.yaml
│       └── generated/                       gocell generate 产物（禁手编辑）
│
├── actors.yaml
│
├── adapters/                                ⚠ 仅当 GoCell 没提供该外部系统
│   ├── apns/                                跨 cell 复用的项目自建 adapter
│   └── salesforce/
│
├── internal/                                ⚠ Go 惯例：项目内跨 cell 共享
│   ├── middleware/ types/ testutil/
│
├── pkg/                                     ⚠ 一般不需要；仅当 export 给外部时用
│
├── tests/integration/
├── fixtures/
├── tools/archtest/                          ⚠ 项目专属 archtest（DX-4 后可继承 gocell 规则）
├── generated/
└── docs/
    ├── architecture/ reviews/ plans/ product/roadmap/
```

**禁止**：`kernel/` `runtime/`；`cells/` 直接 import `adapters/`

### 6.3 形态 C · 可复用 cell 集合（中间层）

```
gocell-zerotrust/                            module: github.com/myorg/gocell-zerotrust
├── go.mod                                   require github.com/ghbvf/gocell vX.Y.Z
├── README.md
├── docs/consumer-guide.md
│
├── cells/                                   ★ 一级平铺
│   ├── devicecore/
│   ├── policycontrol/
│   ├── riskscore/
│   └── compliancecore/
│
├── contracts/                               ★ 集合自带契约
│   ├── http/ event/ shared/
│
├── journeys/                                ⚠ 模板供消费方参考
│
├── adapters/                                ⚠ 集合专属 adapter（OPA wrapper 等）
│
├── internal/                                ⚠ 集合内部共享
│
└── pkg/                                     ⚠ 公开扩展点
```

**禁止**：`cmd/` `main.go` `assemblies/` `kernel/` `runtime/`

### 6.4 三条强制规则（archtest 守护）

无论哪种形态：

1. **`cells/{cell}/` 生产代码不 import 任何 `adapters/`**
2. **`adapters/{x}/` 不 import 任何 `cells/{cell}/`**（除 `cells/{cell}/internal/adapters/{x}/` 例外）
3. **`cmd/{bundle}/` 是唯一同时 import `cells/` + `adapters/` 的地方**

### 6.5 调用链（依赖反转）

```
┌─────────────────────────────────────────────────────────────┐
│  cmd/{bundle}/main.go        ← composition root             │
│    pgRepo := postgres.NewDeviceRepo(pool)                   │
│    cell   := devicecore.New(devicecore.WithRepo(pgRepo))    │
└─────────────────────────────────────────────────────────────┘
                               ↓ 注入
┌─────────────────────────────────────────────────────────────┐
│  cells/devicecore/cell.go                                   │
│    type DeviceCore struct { repo ports.DeviceRepo }         │
└─────────────────────────────────────────────────────────────┘
                               ↑
┌─────────────────────────────────────────────────────────────┐
│  cells/devicecore/internal/ports/repo.go                    │
│    type DeviceRepo interface { Save(...) error; ... }       │
└─────────────────────────────────────────────────────────────┘
                               ↑ 实现
┌─────────────────────────────────────────────────────────────┐
│  github.com/ghbvf/gocell/adapters/postgres                  │
│    var _ ports.DeviceRepo = (*DeviceRepo)(nil)              │
└─────────────────────────────────────────────────────────────┘
```

### 6.6 自建 adapter 决策树

```
你要接的外部系统是什么？
│
├─ GoCell adapters/ 已有？
│   └─ ✅ 直接 import "github.com/ghbvf/gocell/adapters/{x}"
│
├─ GoCell 没有但 kernel/runtime 已定义接口（如 outbox.Publisher）？
│   └─ 情况 A：你写 adapter 实现 gocell 接口
│       ├─ 跨 cell 复用？  → adapters/{x}/
│       └─ 单 cell 专属？  → cells/{cell}/internal/adapters/{x}/
│
└─ GoCell 完全没接口（业务专属，如 APNs / Salesforce）？
    └─ 情况 B：在 cell internal/ports/ 自定义接口 + 写 adapter
        ├─ 跨 cell 复用？  → 接口提到 internal/ports/，adapter 放 adapters/{x}/
        └─ 单 cell 专属？  → 接口在 cell internal/ports/，adapter 在 cells/{cell}/internal/adapters/{x}/
```
