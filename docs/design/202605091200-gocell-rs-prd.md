# GoCell-RS PRD：Cell-Native Rust 框架（垂直切片版）

> 状态：Draft / Discussion
> 作者：Architecture WG
> 日期：2026-05-09
> 范围：把 GoCell（Go）的核心 IP 移植到 Rust，按 L0→L4 一致性等级做 6 个端到端可运行的垂直切片
> 非范围：替换 Go 版 GoCell；polyglot 互操作；WASM/edge 场景

---

## 0. 摘要（TL;DR）

GoCell-RS 是 GoCell 的 Rust 平行实现，目标是验证：**Rust type system + proc macro + cargo workspace 能否吃掉 GoCell ~50% 的治理工作量（archtest 74 → ~17、cell_gen.go → derive、errcode 三层 → type-safe）**。

不一次性移植全部 23 个 kernel 子包 + 12 个 adapter，而是用 **6 个垂直切片（M0–M5）** 渐进交付，每个切片都是"能跑、能测、能 demo"的完整应用，覆盖一致性等级中的 L0/L1 → L2 → L3 → L4 → 治理 → 工程化。

通过条件：M0+M1 PoC 通过 → 决定是否全面投入；任一 PoC 失败 → 退回 Go 或重新评估。

---

## 1. 背景与动机

### 1.1 为何需要

GoCell 在 Go 生态填补的 gap（type system 弱 / 无 macro / error 抽象薄 / outbox 无标准）在 Rust 里有 50–70% 被语言能力自然消化。但**剩余 30–50% 的领域 IP**（cell 模型、一致性等级、outbox + 幂等、contract 治理闭环）在 Rust 生态**无对应物**——Restate 是 workflow runtime，Dapr 是 sidecar，actix/ractor 是 actor 框架，都不覆盖 cell-native 治理。

GoCell-RS 的真实独特价值：

1. **把 cell + 一致性等级（L0–L4）作为一等公民**带进 Rust 生态
2. **proc macro + dylint 替代 archtest**，证明"治理可以编译期 100% 锁定"
3. **errcode 三层 redaction（Message/Details/Internal）由 type system 强制**，PII 泄漏在编译期不可表达

### 1.2 非目标

- ❌ 替代 Go 版 GoCell（Go 版继续主线）
- ❌ 双向 wire 兼容（contract.yaml 复用，但不保证 Go/Rust 二进制互调）
- ❌ 一次性 1:1 移植（74 archtest 全搬、12 adapter 全实现）
- ❌ WASM / no_std / edge 场景
- ❌ 性能基准 vs Go（验证设计可行性，不是性能比赛）

## 3. 垂直切片路线图

### 设计原则

1. **每个切片都是端到端可运行的 demo 应用**——不允许"只交付 trait 不交付实现"
2. **能力按 L0→L4 顺序解锁**——切片 N 不依赖切片 N+1 的能力
3. **明确"不引入"边界**——避免范围蔓延
4. **每个切片有可执行验收脚本**——`cargo run --example {milestone}-acceptance` 即通过

### 切片总览

| 切片 | 主题 | 一致性等级 | Demo 应用 | 工作量 |
|---|---|---|---|---|
| **M0** | HelloCell | L0, L1 | `noteshelf` 单 cell CRUD | 1.5 mo |
| **M1** | NoteEvents | + L2 | `noteshelf` 加 outbox 发事件 | 1.5 mo |
| **M2** | NoteSearch | + L3 | 加 `searchcore` cell 订阅事件做投影 | 1.5 mo |
| **M3** | DeviceCommands | + L4 | 加 `devicecore` cell 下发命令 | 1.5 mo |
| **M4** | Governance | — | proc macro + dylint + CLI 替换 yaml/archtest | 2.0 mo |
| **M5** | Hardening | — | otel/prom/vault/redaction 接入 + docker-compose | 1.5 mo |
| **合计** | | | | **~10 mo** |

每个切片在下面单独展开。

### M0：HelloCell（L0 + L1）

#### 目标

证明**Rust 能跑 GoCell 的最小心智模型**：cell + slice + contract + bootstrap + http + sqlx，全部在一个 cargo workspace 里，提供端到端 CRUD。

#### Demo 应用：`noteshelf`

单 cell `noteshelf`，3 个 slice：

| Slice | 一致性 | 能力 |
|---|---|---|
| `noterender` | L0 LocalOnly | 纯函数：markdown → html，无 DB |
| `notecreate` | L1 LocalTx | POST `/api/v1/notes`，写本地 PG |
| `notequery` | L1 LocalTx | GET `/api/v1/notes/{id}`、GET `/api/v1/notes?cursor=` 列表 |

#### 引入能力（GoCell 包对照）

| GoCell 包 | Rust 实现 | 备注 |
|---|---|---|
| kernel/cell + kernel/slice | `gocell-kernel::cell` trait + assoc types | M0 用 `cell.toml` + 手动 register；M4 改 derive |
| kernel/contract | trait + yaml schema | contract.yaml 直接复用 GoCell 格式 |
| kernel/lifecycle | `tokio_util::CancellationToken` | |
| kernel/registry / assembly | `Arc<RwLock<HashMap>>` + `JoinSet` | |
| kernel/clock | `trait Clock { Real, Frozen }` | |
| runtime/http | `axum` + `tower` | |
| runtime/bootstrap | `App::run()` + tokio main | |
| runtime/config | `figment` + yaml + env | |
| adapters/postgres | `sqlx::PgPool` | |
| pkg/errcode | 自建（D 类） | 三层 Message/Details/Internal 由 type 表达 |
| pkg/redaction | 自建（D 类） | regex sealed key list |
| pkg/idutil / httputil / validation / aeadutil | `uuid` / axum helpers / `validator` / RustCrypto | |

#### 关键技术决策

1. **cell 声明形式**：M0 用 `cell.toml`（serde 解析）+ 在 `main.rs` 手动 `app.register::<NoteshelfCell>()`。**M4 才换成 `#[cell(id="noteshelf", consistency=L1)]` derive 宏。** 这样 M0–M3 的设计漂移不会反复改 proc macro。
2. **DB 层**：`sqlx::query!` 编译期 SQL 校验。CI 必须 sqlx offline mode 落库 schema cache。
3. **错误模型 PoC**：`errcode::Error { kind, message: &'static str, details: Vec<Field>, internal: Option<Box<dyn Debug + Send>> }`，`internal` 字段不可 `Serialize`，wire 层 `impl Serialize` 检查 status 5xx 时 strip details。
4. **测试基线**：每个 slice 有 unit + integration（testcontainers + PG）。L0 slice 必须无任何 IO mock。

#### 不引入（边界）

- ❌ outbox / 事件
- ❌ 跨 cell 通信
- ❌ proc macro
- ❌ dylint / archtest
- ❌ otel / prometheus（只用 `tracing` 输出 stdout）
- ❌ vault / oidc（M0 用 hard-coded JWT secret）
- ❌ ratelimit / circuitbreaker

#### 验收标准

```bash
cargo run -p noteshelf
curl -X POST http://localhost:8080/api/v1/notes -d '{"title":"x","body":"# y"}'   # 201
curl http://localhost:8080/api/v1/notes/{id}                                       # 200
curl http://localhost:8080/healthz                                                  # 200
curl http://localhost:8080/readyz                                                   # 200
RUST_LOG=info cargo test --workspace                                                # all green
```

- `tracing` 输出含 `cell="noteshelf"` 字段
- `errcode` 故意构造 5xx → 响应 `details` 为空、`internal` 不出现在 wire
- PG 断连 → readyz 返回 503，healthz 仍 200

### M1：NoteEvents（L2 OutboxFact）

#### 目标

证明 **outbox + 幂等 Claim/Commit/Release + DLX** 在 Rust 里能干净表达，且 lease fencing token 由 type system 守 CAS。

#### Demo 应用：`noteshelf` 加 slice `noteannounce`

`notecreate` slice 升级为 L2：在同一本地事务内写 note 表 + outbox 表，relay 异步发 `note.created.v1` 到 RabbitMQ。

#### 引入能力

| GoCell 包 | Rust 实现 |
|---|---|
| kernel/outbox | `trait OutboxStore { fn enqueue / claim_pending / mark_published / ... }` |
| kernel/idempotency | `trait Deduplicator` 两阶段 Claim/Commit/Release |
| kernel/command | 暂不引入（M3） |
| kernel/wrapper | `tracing::instrument` + tower middleware |
| runtime/outbox | Store/Relay/Publisher 编排 |
| adapters/postgres outbox | sqlx 实现 OutboxStore，含 lease fencing CAS |
| adapters/rabbitmq | `lapin` Publisher，DLX 配置 |
| pkg/redaction（last_error sanitize） | 复用 M0 |

#### 关键技术决策

1. **Lease fencing token 用 type 强制**：
   ```rust
   #[derive(Copy, Clone)]
   pub struct LeaseId(uuid::Uuid);
   // ClaimPending 返回 (Entry, LeaseId)
   // mark_published(entry_id, lease_id) — SQL CAS 用 lease_id
   ```
   handler 不接触 LeaseId（编译期不可见），等同 archtest `OUTBOX-LEASE-ID-CAS-01` 守卫。
2. **HandleResult / Disposition 用 enum**：
   ```rust
   pub enum Disposition { Ack, Requeue, Reject }
   pub struct HandleResult { disposition: Disposition, err: Option<HandlerError> }
   ```
   `HandleResult::default()` 不实现 `Default`（强制显式声明，避免"零值是 invalid Disposition"问题）。
3. **PermanentError 是标签不是触发器**：`HandlerError::Permanent(e)` 仅用于日志/metric 区分，路由由 `Disposition` 决定（与 GoCell ADR-202605031900 对齐）。
4. **DispatchError redaction**：sanitize 层在 `OutboxStore::mark_retry` 写入 last_error 前调用，单源走 `gocell-redaction`。

#### 不引入

- ❌ 跨 cell consumer（M2）
- ❌ 多 outbox 实现（只 PG）
- ❌ 多 broker（只 RabbitMQ）
- ❌ proc macro
- ❌ kafka / nats

#### 验收标准

```bash
docker compose up -d postgres rabbitmq
cargo run -p noteshelf-events
curl -X POST .../notes -d '...'                  # 201
# 验证 outbox 表有 1 条 pending → 几秒后变 published
psql -c "select status, lease_id from outbox_entries"
# 验证 RabbitMQ 收到 note.created.v1
rabbitmqadmin get queue=note.created.v1
# 故意 kill relay 中 worker → 重启后 lease 过期，新 worker 接管
```

- 单元测试覆盖：lease 过期、CAS 守卫旧 worker mark 失败、DLX 路由
- 集成测试：handler 返回 Reject → 消息进 DLX queue
- 治理（M0 已有 dylint 在 M4 引入，本切片人工 review）：handler 函数签名不能拿到 LeaseId

### M2：NoteSearch（L3 WorkflowEventual）

#### 目标

证明**跨 cell 通信编译期可锁定** + **projection cell 模式**在 Rust 里能干净表达，跨 cell 直接 import 由 cargo workspace visibility 阻断。

#### Demo 应用：加 `searchcore` cell

新 cell `searchcore`，1 个 slice `noteindex`（L3 WorkflowEventual）：订阅 `note.created.v1`，写入本地 `search_notes` 投影表（含 tsvector），暴露 `GET /api/v1/search?q=` 全文搜索。

#### 引入能力

| GoCell 包 | Rust 实现 |
|---|---|
| runtime/eventrouter | `trait EventRouter` + drain RegistrySnapshot |
| ConsumerBase + middleware | `SubscriberWithMiddleware<EntryHandler>` |
| kernel/wrapper.WrapConsumer | `WrapConsumer(handler).with_redactor()` |
| pkg/ctxkeys (cell label) | tracing span fields |
| 跨 cell contract 三处声明对齐 | M2 仍人工，M4 由 dylint + proc macro 检查 |

#### 关键技术决策

1. **跨 cell 隔离用 cargo workspace member 边界**：
   ```
   crates/cells/noteshelf/   pub use ...
   crates/cells/searchcore/  pub use ...
   ```
   `searchcore` 的 `Cargo.toml` 不允许 `noteshelf = { path = ... }` 依赖。**只能依赖 `gocell-contracts` crate**（contract types 唯一来源）。等同 archtest `CELL-INIT-NO-RUNTIME-IMPORT` 由 cargo 天然锁定。
2. **Subscription 注册走 builder**：
   ```rust
   impl Cell for SearchcoreCell {
       fn init(&self, reg: &mut Registry) -> Result<(), Error> {
           reg.subscribe::<NoteCreatedV1>("searchcore", self.handler.clone());
           Ok(())
       }
   }
   ```
3. **handler 签名不接触 Settlement**：
   ```rust
   pub type EntryHandler<E> = Arc<
       dyn for<'a> Fn(&'a Context, Entry<E>) -> BoxFuture<'a, HandleResult>
       + Send + Sync,
   >;
   ```
   Settlement 注入由 `SubscriberWithMiddleware` 内部完成（与 ADR-202605031900 对齐）。
4. **contract.yaml 三处对齐**：M2 通过 `gocell-cli validate`（M4 才完整，M2 用 stub 版）人工跑过，**故意 break 三处之一 → cargo build 失败**。这是 M2 验收的关键。

#### 不引入

- ❌ L4 设备命令
- ❌ 多投影 store（只 PG）
- ❌ proc macro
- ❌ 实时 search（投影最终一致即可）

#### 验收标准

```bash
docker compose up -d
cargo run -p noteshelf-events &
cargo run -p searchcore &
curl -X POST .../notes -d '{"title":"rust","body":"..."}'
sleep 1
curl http://localhost:8081/api/v1/search?q=rust    # 返回投影 hit
# 故意把 searchcore 的 contractUsages 删掉一项 → cargo run 失败：
#   "subscription declared but contract usage missing for note.created.v1"
```

- 集成测试：发布 → 跨进程消费 → 投影建立
- 故意在 `searchcore/src/lib.rs` 加 `use noteshelf::*` → `cargo build` 报 `crate not found`
- DLX 路由：handler 返回 Reject → 投影不写 + DLX 有消息

### M3：DeviceCommands（L4 DeviceLatent）

#### 目标

证明**长延迟单播命令 + ack 回执**模式在 Rust 里能跑通，覆盖 GoCell 一致性等级最后一级。

#### Demo 应用：加 `devicecore` cell + 模拟设备 client

新 cell `devicecore`，2 个 slice：

| Slice | 一致性 | 能力 |
|---|---|---|
| `commandissue` | L4 | POST `/api/v1/devices/{id}/commands`，命令落 queue + 写 outbox |
| `commandack` | L4 | 设备 ack 回调 PUT `/api/v1/commands/{cid}/ack`，更新状态 |

设备模拟器：独立 binary `device-sim`，连 WebSocket（adapter/websocket）接命令，1–10s 后随机 ack。

#### 引入能力

| GoCell 包 | Rust 实现 |
|---|---|
| kernel/command | `trait CommandWriter / Reader` 单播队列 |
| runtime/command | Dispatcher + ReplyRouter |
| runtime/websocket + adapters/websocket | `axum::extract::ws` + `tokio-tungstenite` 客户端 |
| 长延迟超时 | `tokio::time::timeout` + retry budget |
| 命令幂等 | 复用 M1 的 Claimer |

#### 关键技术决策

1. **Command 与 Event 分离的语义在 type 上区分**：
   ```rust
   pub trait Command { type Reply; const KIND: &'static str = "command"; }
   pub trait Event { type Payload; const KIND: &'static str = "event"; }
   ```
   `OutboxEntry<C: Command>` 与 `OutboxEntry<E: Event>` 是不同 monomorphized 类型，编译期不可混用。
2. **Reply 路由用 correlation id**：command_id (UUID) 是 fencing token，ack 必带回；过期由 background sweeper 清理。
3. **设备 client 失联**：consumer base 处理 → Disposition::Requeue 直到 retry budget 耗尽 → Reject 进 DLX。
4. **L4 验证心智**：测试用 `tokio::time::pause()` + frozen Clock 模拟"设备 5 分钟才 ack"。

#### 不引入

- ❌ 设备 fleet 管理 / shadow state
- ❌ MQTT（M3 只 WebSocket）
- ❌ 多设备并发命令调度
- ❌ proc macro

#### 验收标准

```bash
cargo run -p devicecore &
cargo run -p device-sim -- --device-id=dev-01 &
curl -X POST .../devices/dev-01/commands -d '{"action":"reboot"}'
# 设备 sim 收到 → 5s 后 ack
sleep 6
curl .../commands/{cid}    # status: acked
# 故意 kill device-sim → 命令 timeout → retry budget 耗尽 → DLX
```

- 集成测试：frozen clock 下时间快进 60s 验证超时路径
- 验证 command_id correlation 不重复 ack
- ack 收到后 outbox 写"command.completed.v1" 给审计

### M4：Governance（proc macro + dylint + CLI）

#### 目标

把 M0–M3 的 yaml + 手动 register + 人工 review 全部替换成**编译期/CI 期强制**：proc macro + dylint + gocell-cli。**核心命题：能否把 GoCell 74 个 archtest 收敛到 ≤ 20 个 dylint。**

#### Demo

不是新业务应用，是把 M0–M3 全部"治理化"。改造点：

- `cell.toml` → `#[derive(Cell)] #[cell(id="noteshelf", consistency=L1)]`
- contract.yaml → build.rs + `schemars` 派生 Rust types（gocell-derive 提供 `#[contract]`）
- `app.register::<NoteshelfCell>()` 手工 → proc macro `register_cells!` 宏自动展开

#### 引入能力

| 工具 | Rust crate |
|---|---|
| `gocell-derive` | proc-macro2 + syn + quote + proc-macro-error |
| `gocell-codegen` | schemars + prettyplease + serde_yaml |
| `gocell-cli` (validate / scaffold / generate / check / verify / graph) | clap + 上面工具组合 |
| `gocell-lints` (dylint workspace) | dylint_linting + clippy_utils |

#### 关键技术决策

1. **derive 宏作用域**：
   ```rust
   #[derive(Cell)]
   #[cell(
       id = "noteshelf",
       consistency = "L1",
       owner = "platform-team",
       schema = "primary",
   )]
   pub struct NoteshelfCell { /* ... */ }
   ```
   宏自动展开 `impl Cell for NoteshelfCell`，并生成 `__GOCELL_REGISTRATION_NOTESHELF` 静态项（链接期发现）。
2. **contract.yaml 编译期摄入**：
   ```rust
   gocell_contracts::include_contract!("event/note/created/v1/contract.yaml");
   // 展开为 pub struct NoteCreatedV1 { ... } + impl Event for NoteCreatedV1
   ```
   build.rs 不引入（避免编译期 IO 复杂度）；用 `include_str!` + proc macro 处理。
3. **dylint 规则集（目标 ≤ 20 条，对应 GoCell 74 archtest）**：
   | 类别 | 规则 | 替代 GoCell archtest |
   |---|---|---|
   | M | `gocell_message_const_literal` | MESSAGE-CONST-LITERAL-01 |
   | M | `gocell_details_field_only` | DETAILS-SLOG-ATTR-01 |
   | M | `gocell_internal_no_serialize` | (新, type-safe 兜底) |
   | O | `gocell_outbox_lease_not_in_handler` | OUTBOX-LEASE-ID-CAS-01 |
   | O | `gocell_handler_no_settlement` | HANDLERESULT-NO-RECEIPT-FIELD-01 |
   | P | `gocell_panic_whitelisted` | PANIC-REGISTERED-01 |
   | C | `gocell_contract_usage_aligned` | ADV-06 / VERIFY-01 |
   | R | `gocell_redis_keynamespace` | REDIS-KEY-NAMESPACE-01 |
   | H | `gocell_http_metrics_cell_label` | CTXSOURCE / RUNTIME-SENTINEL |
   | T | `gocell_test_time_clock` | (test time discipline) |
   | … | (~10 条更多) | |
   **目标：剩余 50+ archtest 由 type system / cargo workspace / proc macro 100% 消化。**
4. **proc macro 错误信息基线**：所有错误必须 `proc-macro-error` 输出，含 source span 指向 yaml 行号。`trybuild` 测试覆盖每条诊断。

#### 不引入

- ❌ Backstage / GraphQL export（不在 PoC 范围）
- ❌ 自定义 IDE LSP 插件
- ❌ 完整 90% 的 archtest 覆盖（PoC 阶段 50% 即可）

#### 验收标准

```bash
cargo install --path tools/gocell-cli
gocell scaffold cell mycell --consistency L1 --owner platform
cd cells/mycell && cargo build              # 通过

# 故意写违规
echo 'errcode::new(ErrInternal, format!("user {} bad", id))' >> src/lib.rs
cargo dylint --all                          # FAIL: gocell_message_const_literal

# 故意删 contract usage
# 删 cell.rs 里 #[cell(consumes=["note.created.v1"])] 一项
cargo build                                 # FAIL: subscription declared but cell does not consume

# 故意把 internal 字段加 derive(Serialize)
cargo build                                 # FAIL: gocell_internal_no_serialize
```

- M0–M3 全部应用在 M4 完成后**仅修改 cell 声明形式**（yaml → derive），业务逻辑不变，回归测试全绿
- archtest 剩余规则数 ≤ 20，单条规则 ≤ 200 LOC dylint 实现
- `gocell graph` 输出 dot 文件可视化 cell 依赖

### M5：Hardening（observability + 安全 + 部署）

#### 目标

把 M0–M4 接入生产级可观测性 + 安全栈，证明 **PII 不泄漏 + 跨 cell trace 串联** 在 Rust 里端到端可达。

#### Demo

`docker-compose.yml` 启动：noteshelf + searchcore + devicecore + postgres + rabbitmq + redis + jaeger + prometheus + grafana + vault。完整业务路径走通后，从 Jaeger UI 看到一条 trace 跨 3 个 cell。

#### 引入能力

| GoCell 包 | Rust 实现 |
|---|---|
| adapters/otel | `opentelemetry-otlp` + jaeger exporter |
| adapters/prometheus | `metrics` + `metrics-exporter-prometheus`，cell label 由 tracing span 派生 |
| adapters/vault | `vaultrs` Transit |
| adapters/oidc | `openidconnect` |
| adapters/redis (idempotency / cache / nonce) | `fred` |
| pkg/redaction | 端到端串联：tracing layer + outbox last_error + log handler |
| pkg/securecookie | `cookie` + RustCrypto AEAD |
| runtime/auth (JWT + service token) | `jsonwebtoken` + tower layer |
| adapters/ratelimit | `governor` |
| adapters/circuitbreaker | `failsafe` 或薄自建 |

#### 关键技术决策

1. **cell label 单源**：`tracing::Span` 当前 span 的 `cell` field 是唯一来源，metrics 中间件读这个 field（与 GoCell `ctxkeys.CellID` 同语义）。
2. **PII redaction fail-closed**：
   ```rust
   pub struct Redactor { regex_set: SealedRegexSet }
   // tracing layer + opentelemetry SpanProcessor + outbox last_error 三处共享同一 Redactor 实例
   ```
   故意打 `tracing::error!("password=secret123 user=alice")` → jaeger / loki / outbox last_error 全部看到 `password=<REDACTED> user=<REDACTED>`。
3. **metrics cell label 默认过滤 `_runtime`**：grafana dashboard 模板含 `{cell!="_runtime"}`，告警 PromQL 同步。
4. **Vault Transit 替代本地 AES**：M0 hard-coded key → M5 `KeyProvider` trait 走 Vault。

#### 不引入

- ❌ Kubernetes operator（独立 PoC）
- ❌ 多区域部署
- ❌ 自定义 grafana dashboard 全集（提供 1 个 demo dashboard）
- ❌ 完整 SOC2 audit log 流水

#### 验收标准

```bash
docker compose up -d
# 业务路径
curl -X POST .../notes -d '{"title":"x","body":"password=secret123"}'
# Jaeger UI 看 trace
open http://localhost:16686
# 一条 trace 跨 noteshelf → rabbitmq → searchcore → 3 个 service span
# trace 内 attributes 含 cell="noteshelf" / "searchcore"
# log message body 中 "password=secret123" 已被 redact

# Prometheus 查询
curl http://localhost:9090/api/v1/query?query=http_requests_total{cell="noteshelf"}
curl http://localhost:9090/api/v1/query?query=http_requests_total{cell="_runtime"}  # /healthz 等

# 故意打 PII log
RUST_LOG=info cargo run ... 2>&1 | grep -c "password=<REDACTED>"   # ≥ 1
RUST_LOG=info cargo run ... 2>&1 | grep -c "secret123"             # = 0
```

- 跨 cell trace 在 Jaeger 可见，深度 ≥ 3
- prometheus `http_requests_total{cell=...}` 标签正确
- redaction 端到端无遗漏（tracing + log + outbox last_error）
- ratelimit / circuitbreaker 触发后正确返回 429 / 503

## 5. 度量与验收

### 5.1 PoC 通过判据（M0 + M1 完成后评估）

**必要条件（全部满足）**：

1. ✅ `noteshelf` 端到端 CRUD 工作，L0/L1/L2 一致性等级语义在代码里清晰可见
2. ✅ `errcode` 三层结构 PII 在编译期不可表达（`Internal` 不能 `Serialize`，故意尝试 → 编译失败）
3. ✅ outbox lease fencing token type-safe，handler 拿不到 LeaseId
4. ✅ 单 cell 工作量 ≤ Go 版的 70%（用代码行 + 文件数粗算）
5. ✅ 文档 / 错误信息可读性不劣于 Go 版

**充分条件（≥ 2 项）**：

1. 🎯 archtest 数量在 M4 验证可降至 ≤ 20（M0/M1 期间用 mock 评估）
2. 🎯 proc macro 调试体验"可接受"——团队 ≥ 2 人不抗拒
3. 🎯 编译时间在中型工作区（30 cell）≤ 90s 增量

### 5.2 失败判据（任一即停）

- ❌ async trait dyn-safe 边界导致 registry / event router 必须放弃 `dyn` 使用 generic monomorphization，跨 cell 类型爆炸
- ❌ `errcode` 三层无法在不引入 `unsafe` 的前提下做到编译期 strip
- ❌ proc macro 错误信息只能给"in this macro expansion"无 source span，业务作者无法定位
- ❌ M0 完成后总体工作量超出预算 50%

### 5.3 长期度量（M5 之后）

| 指标 | 目标 |
|---|---|
| archtest dylint 规则数 | ≤ 20 |
| archtest 由 type system 消化率 | ≥ 70% |
| 单 cell scaffold 到首次 200 OK 时间 | ≤ 5 min |
| 跨 cell trace 端到端延迟 | ≤ 50ms p99（demo 环境） |
| PII 泄漏测试用例覆盖 | ≥ 95% |
