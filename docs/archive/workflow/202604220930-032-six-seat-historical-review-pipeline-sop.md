# 六席位历史审查流水线实施SOP（v1）

> 日期: 2026-04-22
> 目标: 将“历史审查提取 -> 分类汇总 -> 领域分析 -> 分层分析 -> TOP10深挖”固化为可重复执行流程。
> 硬约束: 严格满足六席位并行审查与 3+ 开源项目对标证据门禁。

---

## 1. 适用范围

- 历史审查资料复盘
- backlog/tech-debt 聚合治理
- PR 前置风险盘点
- 版本发布前的系统性风险盘点

输入源默认白名单:
- docs/reviews
- docs/backlog.md
- docs/tech-debt-registry.md
- specs
- bak/docs/reviews
- VS Code Copilot debug-logs（增强来源）

---

## 2. 固定角色（六席位）

1. 架构席位: 关注分层、契约边界、调用链、耦合方向。
2. 安全席位: 关注认证鉴权、越权路径、重放与数据暴露。
3. 测试回归席位: 关注覆盖率、回归缺口、旅程可验证性。
4. 运维部署席位: 关注启动路径、配置生命周期、可观测性、故障域。
5. 可维护性/DX席位: 关注复杂度、可读性、重复逻辑、治理一致性。
6. 产品体验席位: 关注用户路径、错误可理解性、交互与文档可达性。

规则:
- 六席位独立产出，先各自结论，后统一归并。
- 归并按根因簇，不按文件表象。

---

## 3. 分阶段执行

### Phase A: 数据定义与基线建立（阻塞后续）

输入:
- 输入源白名单中的原始文档

动作:
1. 使用 templates/review-finding-normalized-schema.yaml 作为统一字段模型。
2. 建立证据索引表，确保每条结论可回溯到来源路径和行号。
3. 记录不可解析项和歧义项。

输出:
- A1 基线报告（覆盖率、解析成功率、歧义项）

验收:
- 关键字段完整率 >= 95%
- 证据可追溯率 = 100%

### Phase B: 常见问题提取（对应步骤1）

输入:
- Phase A 统一模型

动作:
1. 六席位并行抽取 Raw Findings。
2. 每席位独立输出 Seat Digest。
3. 汇总层做根因归并，形成 Common Issues Report。

输出:
- 常见问题报告 V1
- 席位摘要集

验收:
- 六席位完整率 = 100%
- 根因归并覆盖率 >= 90%

### Phase C: 分类/频次/处置/对标状态汇总（对应步骤2）

输入:
- Phase B 常见问题主表

动作:
1. 多维分类: domain/layer/severity/complexity/rootCauseCluster。
2. 频次统计: 总频次 + 近窗口频次 + 复发频次。
3. 处置矩阵: Now / This Iteration / Backlog。
4. 对标状态矩阵: complete / pending / insufficient。

输出:
- 分类汇总报告 V1

验收:
- 分层归属完整率 >= 98%
- 领域归属完整率 >= 98%

### Phase D1: 领域六席位分析（对应步骤3）

动作:
1. 按领域切片，每个领域启动六席位并行分析。
2. 输出领域风险图谱与优先级清单。

输出:
- 领域报告集

### Phase D2: 分层六席位分析（对应步骤4）

动作:
1. 按分层切片（kernel/runtime/cells/adapters/contracts/pkg/cmd/examples）。
2. 每层输出层内风险、跨层耦合和治理建议。

输出:
- 分层报告集

### Phase D3: TOP10 六席位深挖（对应步骤5）

动作:
1. 以风险分选取 TOP10。
2. 每个问题执行六席位并行深挖。
3. 输出双方案: 最小修复A + 彻底修复B。

输出:
- TOP10 深挖报告
- 路线图

TOP10 推荐评分:
- score = severityWeight + frequencyWeight + blastRadiusWeight + regressionRiskWeight - mitigationReadinessBonus

### Phase E: 开源对标证据门禁（可并行，最终发布前必须完成）

动作:
1. 先按主题聚类需要外部论证的问题。
2. 每个主题至少并行 3 个研究子agent。
3. 优先项目池: Kubernetes, Uber fx, Kratos, go-zero, go-micro, Watermill。

门禁:
- 少于3项目证据，禁止使用“最佳实践”结论。
- 项目间冲突时必须给出权衡，不得强行单结论。

### Phase F: 汇总交付

输出包:
1. SOP执行包（输入/输出/动作/验收/失败回退）
2. 项目管理包（里程碑/工时/关键路径/风险）
3. 最终总报告

---

## 4. 并行与依赖

可并行:
- Phase B 六席位
- Phase D 每轮内六席位
- Phase E 每主题 3+ 对标研究

必须串行:
- A -> B -> C

发布门禁:
- D 可与 E 并行推进，但最终报告发布必须等待 E 完成。

---

## 5. 失败回退

1. 文档解析失败: 进入人工复核池并打标 unresolved。
2. 席位冲突: 不覆盖少数意见，进入冲突仲裁表。
3. 对标不足: 明确标注 insufficient，不给最佳实践结论。

---

## 6. 执行清单（可复制）

1. 准备输入源并生成证据索引。
2. 六席位并行输出 Raw Findings。
3. 按根因簇归并并形成常见问题报告。
4. 输出分类/频次/处置/对标状态矩阵。
5. 按领域执行六席位分析并生成领域报告。
6. 按分层执行六席位分析并生成分层报告。
7. 计算TOP10并执行六席位深挖。
8. 对每个主题补齐 3+ 开源项目对标证据。
9. 生成最终综合报告与项目排期。
