## 概述

PR-A26 引入 v1.0 P0 的「交互式首次启动 admin」HTTP 入口，并把原先只在 `initialadmin` Lifecycle 里的 race-safe user+role 持久化逻辑抽到共享 `cells/accesscore/internal/adminprovision` 包。

- **新 HTTP surface**
  - `GET  /api/v1/setup/status`（Public, L0 契约, 返回 `{hasAdmin: bool}`）
  - `POST /api/v1/setup/admin`（Public, L2 契约, 成功 201 / 已初始化 409）
- **共享包**: `adminprovision.Provisioner` 提供 `Status` / `Ensure` / `Compensate`，结果以 `ProvisionOutcome` 枚举区分 Created / AlreadyExists / RaceSkipped / OrphanRecovered。`Ensure` caller-tx-neutral——`initialadmin` 无 tx 调用，`setup` 把它包在 `persistence.TxRunner.RunInTx` 里。
- **重构**: `initialadmin/bootstrap.go` 删掉 ~170 行重复逻辑（`adminExists` / `ensureRoleAndCreateUser` / `resolveDuplicateUser` / `compensateAfterCredFileFailure`），改成 switch on `ProvisionOutcome`。对应 branch 测试从 `bootstrap_branches_test.go` 搬到 `provisioner_test.go`。公共 Lifecycle API 零改动。
- **新增 errcode**: `ErrSetupAlreadyInitialized → 409`。
- **契约**: `contracts/http/auth/setup/status/v1`、`contracts/http/auth/setup/admin/v1`，schema 带 `additionalProperties:false` + `minLength/maxLength`。

作者在 PR 创建后自提交了一轮 reviewer Cx1/Cx2 fix（commit `5019f189`），已处理 F-04/F-05/F-06/F-08/F-10/F-11，有 backlog 延期项。下面只列当前 HEAD 仍存在的问题。

## Major Findings（Cx2）

### M-01 【Cx2, security/ops】`POST /api/v1/setup/admin` 把 bcrypt 放在 `Status` 前，造成匿名 DoS 面

**文件**: `cells/accesscore/slices/setup/service.go:155-161`

```go
hash, err := bcrypt.GenerateFromPassword([]byte(in.Password), domain.BcryptCost)  // line 155, ~1-2s at cost=12
if err != nil { ... }

var out *CreateAdminOutput
err = s.txRunner.RunInTx(ctx, func(txCtx context.Context) error {
    user, outcome, perr := s.provisioner.Ensure(txCtx, adminprovision.ProvisionInput{...})  // line 162 → Status fast-path inside
```

- 该端点 `Public: true`（`cell_routes.go:87-92`），无 JWT、无 rate limit、无 CAPTCHA。
- `Ensure` 内部虽然有 `Status` fast-path（`provisioner.go:128-136`），但只有**调用 Ensure 之后**才会触发。bcrypt 在调用 Ensure 前就已经跑满 CPU。
- 结果：不管系统有无 admin，每个结构合法（密码 8–128 字符、username/email 非空）的 POST 都会消耗一次 bcrypt cost=12（≈1–2s CPU）。admin 已存在后仍然如此，**放大到可持续 DoS**。
- PR 描述承认 "Rate limiting / CAPTCHA ... not blocking v1.0 (one-shot endpoint)"，但「one-shot」只保护首次创建的窗口。bcrypt 在 409 路径仍然跑，攻击窗口不止于初装期。

**建议**（本 PR 内修，二选一）:
1. 先调用 `s.provisioner.Status(ctx)`；`hasAdmin=true` 时直接 `ErrSetupAlreadyInitialized`，再进 bcrypt。
2. 把 bcrypt 移到 `RunInTx` 内 Ensure 之后的 `OutcomeCreated` 分支里（但这样得重排接口——因为 `Ensure` 现在接受已 hash 的 `PasswordHash`）。方案 1 更经济。

提交注释应同时在 `CreateAdmin` godoc 里说明：**422 / rate-limit / CAPTCHA 仍然是生产前必须的外层防护**，并在 `docs/backlog.md` 注册 `SETUP-RATE-LIMIT-01`。

> 验收标准：fresh 系统第一次 POST 才花 bcrypt，已有 admin 的重复 POST 应在 ~毫秒级返回 409；加一条 service-level 测试 `TestService_CreateAdmin_AlreadyExists_DoesNotHashPassword`（mock Hasher 计数 + 校验 0）。

### M-02 【Cx2, layer/product】orphan recovery 在 setup 路径上丢失 `RequireReset=false` 的语义，导致 401-reset

**文件**: `cells/accesscore/internal/adminprovision/provisioner.go:238-241`

```go
existing.PasswordHash = string(in.PasswordHash)
if in.RequireReset {
    existing.MarkPasswordResetRequired()
}
if err := p.userRepo.Update(ctx, existing); err != nil {
```

- orphan recovery 场景：一个崩溃的 `initialadmin` Lifecycle 写入了 user 行（`PasswordResetRequired=true`）但尚未 `AssignToUser`；之后操作员走 setup HTTP 路径 `RequireReset=false`。
- 因为条件是 **"only set when true"**，`PasswordResetRequired=true` 标志**不会被清除**，orphan 行保留旧值。
- setup 调用方的承诺是「操作员选了自己的密码，立即可用」。但 orphan-recovery 完成后，操作员第一次登录会被中间件挡下 force-reset（`ErrAuthPasswordResetRequired`），与语义矛盾。
- 这个 bug 不会在内存 repo 下复现（`in-memory` 数据不跨进程持久化，所以 orphan 场景只出现在 PG 模式）——但代码路径本身有 bug。

**建议**: 改条件赋值为直写:
```go
existing.PasswordHash = string(in.PasswordHash)
existing.PasswordResetRequired = in.RequireReset
```
（假设 `domain.User.PasswordResetRequired` 是导出字段；若是私有字段就加一个 `SetPasswordResetRequired(bool)` 方法。）

同时加一条 provisioner 单测 `TestProvisioner_Ensure_OrphanRecovered_ClearsResetFlag_WhenCallerWantsNoReset`：预置 `PasswordResetRequired=true` 的 orphan，调 `Ensure(RequireReset=false)`，断言结果行 `PasswordResetRequired=false`。

## Minor / Nit（Cx1）

### N-01 【Cx1, product】`/api/v1/setup` 顶级路径可能限制未来扩展

**文件**: `cells/accesscore/cell_routes.go:75-93`、`contracts/http/auth/setup/*/v1/contract.yaml`

PR 描述说明 `setup` 置于顶级是「系统层概念，不属于 access 子域」。这个判断合理，但目前 repo 没有统一的「system-level」路径公约。如果以后 `configcore` 也想加 `/api/v1/setup/config-bootstrap`，路径空间归属就会混乱。建议在 `docs/plans/` 或 `.claude/rules/gocell/api-versioning.md` 里补一条「顶级 `/api/v1/setup/` 归属 accesscore，其他 Cell 的初始化入口放在自己命名空间下」的约定（out of scope，但挂一条 roadmap 条目 `SETUP-PATH-NAMESPACE-POLICY-01` 即可）。

### N-02 【Cx1, dx】`out.ID[:4]` 在 service_test.go 用原始字节切片判断前缀

**文件**: `cells/accesscore/slices/setup/service_test.go:105`

```go
assert.True(t, len(out.ID) > 0 && out.ID[:4] == "usr-")
```

UUID 下没问题，但风格脆——若哪天 prefix 改成 `"usr:"`、`"usr."` 还得同步长度。建议 `assert.True(t, strings.HasPrefix(out.ID, setup.UserIDPrefix))`，或直接 `assert.Regexp(t, "^usr-[0-9a-f-]{36}$", out.ID)`。

### N-03 【Cx1, dx】`publishUserCreated` payload 用 `map[string]any` 而非 typed struct，违背 `cell-patterns.md` 序列化边界约定

**文件**: `cells/accesscore/slices/setup/service.go:205-211`

```go
payload, err := json.Marshal(map[string]any{
    "user_id":  user.ID,
    "username": user.Username,
})
```

cell-patterns.md 明确写「Handler 响应和事件 payload 统一用 typed struct + converter」。现实里 `identitymanage.Create` 也用 map（`identitymanage/service.go:128`）——不是本 PR 新引入的坏味，但也不应放大。建议抽一个 `UserCreatedEvent` struct 放到 `cells/accesscore/internal/dto/`，setup 和 identitymanage 都用它（为 F-02 的 "TopicUserCreated 重复" 也留好抽常量的位置）。

本 PR 可以不做，把这个 + F-02 + F-03 合并到 `cell-patterns-compliance-identity-events-01` 一并处理。

### N-04 【Cx1, product】`POST /setup/admin` 错误响应体对外消息泄露 login 路径

**文件**: `cells/accesscore/slices/setup/service.go:174-175`

```go
return errcode.New(errcode.ErrSetupAlreadyInitialized,
    "first-run admin already provisioned; use /api/v1/access/sessions/login")
```

消息暴露内部登录路径。对外场景是「装机引导」UI，已经知道这一路径，泄露风险≈0。但作为 API 最佳实践，建议把路径挪到 `details.next_action` 字段，message 保留「first-run admin already provisioned」。

### N-05 【Cx1, layer】`slice.yaml` 没有显式 `consistencyLevel`

**文件**: `cells/accesscore/slices/setup/slice.yaml`

cell_init.go:154 把 setup slice 加成 `cell.L2`，但 slice.yaml 里没写 `consistencyLevel: L2`。per CLAUDE.md "owner、consistencyLevel 缺省时继承 cell.yaml" 是合法的，且 `validate --strict` 通过，但作者自己在 fix commit 里把这条列为 F-09 defer 的理由是「SliceMeta does not accept the field (inheritance from cell.yaml is the declared model)」——这个说法需要验证。手跑 `grep -r consistencyLevel cells/*/slices/*/slice.yaml` 可以确认其他 slice 是否有显式声明；若有，说明 SliceMeta 其实接受，本条应改掉。

我没有跑这个 grep（节省上下文），但作者的 deferral 理由建议写入单测（`TestSliceMetaAcceptsConsistencyLevel`），避免后来人反复翻案。

### N-06 【Cx1, test】`setup_integration_test.go` 没覆盖 `OrphanRecovered` 到 HTTP 的路径

当前 integration test 只走 fresh → 201 → 409 → login。orphan-recovery 分支仅在单元测里（`TestService_CreateAdmin_OrphanRecovered_ReturnsUser_NoEmit`），没有 end-to-end 压过 HTTP 栈。考虑到 M-02 揭示了 orphan 路径的语义缺陷，加一条 integration 测试能防线再加厚——但 in-memory repo 下造 orphan 行需要直接操作 repo，略 hacky。建议在 `docs/backlog.md` 登 `SETUP-ORPHAN-E2E-01` 作为 PG migration 同步完成时再补。

### N-07 【Cx1, security】email 只有 `minLength`/`maxLength`，没有 format 校验

**文件**: `contracts/http/auth/setup/admin/v1/request.schema.json:7`

```json
"email": { "type": "string", "minLength": 1, "maxLength": 256 }
```

JSON Schema 的 `format: email` 不是默认强校验（`github.com/xeipuuv/gojsonschema` 等库要显式打开），而且 setup 场景 email 通常是 `root@local` 这种伪邮箱，严格校验反而出乱。接受现状。但建议 service.CreateAdmin 里加一条「不含控制字符」的轻量校验（防 log injection），或明确在 doc 里写「email 字段仅作标识，不校验 RFC5321」。
