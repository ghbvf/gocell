## CI 变更摘要

本 PR 在 `_build-lint.yml` 中新增可选 Job `os-smoke`，通过新增 reusable workflow 输入参数 `run-os-smoke: boolean（默认 false）` 控制。`ci.yml`（push:develop）和 `pr-check.yml`（pull_request）均将该参数设为 `true`，即 PR 和主干推送都会触发 macOS + Windows 两腿 matrix。Job 使用 `fail-fast: false`，两个 OS 独立运行，任一失败均使整体 workflow run 失败（红灯）。覆盖命令为 `go test -count=1 -timeout 5m`，不带 `-race`，不产出 coverage profile，不上报 Sonar。

## Finding 列表

### C01 — os-smoke 未被设为 required check，无法阻止合并（高优先级）

- **文件**: GitHub 仓库 Branch Protection / Rulesets（当前配置）
- **现状**: `gh api repos/ghbvf/gocell/branches/develop` 返回 `required_status_checks.enforcement_level: "off"`，`rulesets` 为空数组。GitHub 上没有任何 required status check 在守门 develop 分支合并。
- **影响**: 即使 `os-smoke (windows-latest)` 失败，PR 仍然可以被合并（按钮为绿色）。PR 自述"CI: pr-ci / os-smoke (windows-latest) — gate for V-A2/V-A3/V-A4 fixes"在技术层面目前并不成立。
- **建议**: 在 GitHub Settings > Branches > develop 下将 `pr-ci / os-smoke (macos-latest)` 和 `pr-ci / os-smoke (windows-latest)` 加入 Required Status Checks，或通过 Rulesets 的 `required_status_checks` 字段配置。这是本次合并前最关键的手工确认项。

### C02 — os-smoke 缺少 `timeout-minutes`（低优先级）

- **文件**: `.github/workflows/_build-lint.yml`，行 166-180
- **现状**: `os-smoke` Job 本体无 `timeout-minutes`。`go test -timeout 5m` 只保护单次测试执行，不约束整个 Job 的生命周期（含 runner 排队、checkout、setup-go 缓存恢复等）。若 windows-latest runner 队列拥挤，Job 可能挂起 6 小时直到 GitHub 默认超时。
- **对比**: `build-test` 和 `integration-test` 同样缺少 Job 级 `timeout-minutes`，但那是预存问题，不在本 PR 范围内。本 PR 引入了新 Job，应补齐。
- **建议**: 在 `os-smoke` Job 上添加 `timeout-minutes: 20`（checkout + setup-go + 5m test + 缓冲量）。

### C03 — Windows 测试命令未含 `-race`，与 Linux 单元测试不一致（信息）

- **文件**: `.github/workflows/_build-lint.yml`，行 180
- **现状**: `os-smoke` 的测试命令为 `go test -count=1 -timeout 5m ./...`，无 `-race`。Linux `build-test` Job（行 71）运行 `go test -race -coverprofile=coverage.out -coverpkg=./...`。
- **意图符合性**: PR 自述明确将 os-smoke 定位为"跨平台单元冒烟"，不与 Sonar 合并覆盖率，因此不带 `-race` 和 `-cover` 属于有意设计，不违反 CI 一致性要求。Windows 上的 race detector 支持自 Go 1.8 起已稳定，可选升级，但不是阻塞项。
- **建议**: 保持现状可接受；如后续发现 Windows 数据竞争问题，可加 `-race`。

### C04 — Windows 专属代码路径 `credfile_security_windows.go` 的 DACL 逻辑不被 Sonar 覆盖（信息）

- **文件**: `cells/accesscore/initialadmin/credfile_security_windows.go`（本 PR 新增，283 行）
- **现状**: Sonar 分析只消费 Linux 单元覆盖（`coverage.out`）和 Linux 集成覆盖（`coverage-integration.out`）；两者均在 ubuntu-latest 上生成。`credfile_security_windows.go` 中的 `windowsCredfile.EnsureSecureDir` / `SecureNewFile` / `VerifyDACL` 实际被 Windows CI runner 跑过（os-smoke 含 `./cells/accesscore/initialadmin/...`），但对应 coverage profile 不上报 Sonar。
- **影响范围**: Sonar Quality Gate 的"新代码覆盖率 ≥ 80%"门禁无法感知 Windows 专属行数，可能导致 Sonar 把这些行计入"未覆盖"（降低门禁分子），也可能不计入（取决于 Sonar 如何处理平台 build tag 过滤的文件）。PR 自述写明"Coverage NOT merged into Sonar (Linux unit + integration already cover the cross-platform code)"，决策有据，但需要确认 Sonar 实际门禁行为。
- **建议**: 合并前手工在 Sonar 项目页核查 `credfile_security_windows.go` 的行覆盖显示（是否被忽略或被标红），确认 Quality Gate 不因此失败，再批准合并。

### C05 — os-smoke 未声明与 sonarcloud Job 的依赖关系（低优先级）

- **文件**: `.github/workflows/_build-lint.yml`，行 189-191（sonarcloud Job）
- **现状**: `sonarcloud` Job 的 `needs: [build-test, integration-test]`，未包含 `os-smoke`。这意味着即使 os-smoke 失败，sonarcloud Job 仍会正常启动并可能通过，导致绿色的 SonarCloud 结果与红色的 os-smoke 结果并存。
- **影响**: 单纯视觉问题；sonarcloud 本身与 Windows/macOS 测试结果无逻辑依赖，不加入 `needs` 在语义上也可以接受。若希望"Sonar 只在所有烟测通过后运行"，需将 `os-smoke` 加入 `needs`。记录为信息项，不作阻塞。
