## 4. P0 阻断项根因分析

### P0-1: L7-ex — sso-bff 示例启动即崩溃

**根因（数据流）：**
```
main.go:100  query.NewCursorCodec([]byte("sso-bff-audit-cursor-key-32b!!"))
                          ↓ len("sso-bff-audit-cursor-key-32b!!") = 30 bytes
pkg/query/cursor.go:16   if len(key) < minCursorKeyBytes { return nil, error }
                          ↓ error != nil
main.go:102  if err != nil { log.Fatal(err) }  → os.Exit(1)
```

**根因（架构）：** walkthrough_test.go 与 main.go 是完全独立的 bootstrap 逻辑——测试用了正确的 key 长度（32/33 bytes），掩盖了 main.go 的配置错误。CI 只 `go build`，不运行 examples binary。

**影响范围：** 任何试图 `go run ./examples/sso-bff` 或 `docker compose up` 的用户都会遭遇即时崩溃，这是最直接的第一印象伤害。

---

### P0-2: L1 — audit 路由裸注册（安全越权）

**根因（调用链）：**
```
cells/audit-core/cell.go:241
  sub.Handle("GET /entries", http.HandlerFunc(c.queryHandler.HandleQuery))
                             ↑ 直接注册，跳过 auth.Declare
                             
cells/audit-core/slices/auditquery/handler.go:56
  func auditQueryPolicy(r *http.Request) error { /* 存在但从未被调用 */ }

cells/audit-core/slices/auditquery/handler_test.go:243
  securedMux用了auth.Declare → 测试覆盖了正确路径，但生产注册是裸的
```

**根因（架构）：** `auditQueryPolicy` 在 handler.go 中定义但未导出，无法被 cell.go 引用（package 内可见性工作），但 cell.go 没有使用它。auth.Declare 是将 Policy 注入中间件链的唯一路径。

**影响范围：** 任何未登录或低权限用户可访问 `GET /api/v1/audit/entries`，横向读取他人审计记录，违反 OWASP A01。

## 6. 开源对标结论

### 对标 1：路由 Policy 注册（L1/L2）

| 项目 | 注册时机 | 强制手段 |
|------|---------|---------|
| Kubernetes apiserver | 启动构建 handler chain | 全局过滤链包裹（Authn/Authz）；公开端点白名单显式声明 |
| go-chi/chi | 路由注册前 Use() | Group 中间件继承；Walk() 路由树遍历可做启动期审计 |
| Casbin chi-authz | 运行期 Enforce | 策略判定完整，但不防"漏挂中间件" |

**结论：** GoCell 应采用 **声明式路由注解**（RouteDecl 强制携带 Policy/Public/Delegated 三选一）+ **启动期 FinalizePolicyRegistry fail-fast**。参考 chi Walk() 遍历路由树实现合规校验。

### 对标 2：健康检查超时（A21）

| 项目 | 接口签名 | 超时机制 |
|------|---------|---------|
| Kubernetes healthz | `Check(*http.Request) error` | 聚合串行，无统一 deadline |
| go-zero (可用证据) | TimeoutHandler `context.WithTimeout` | handler 级 context 注入 |
| HashiCorp Consul | 独立 goroutine + `select/time.After` | 单检查超时 + 并发周期执行 |

**建议 GoCell 升级方案：**
```go
// 目标接口
type Checker interface {
    Name() string
    Check(ctx context.Context) error
}
// 聚合：全局 2s deadline + errgroup 并发 + 逐项耗时响应
```

### 对标 3：事件 Payload 类型化（S4）

| 项目 | 类型约束方式 | Schema 共享 | Evolution |
|------|------------|-----------|----------|
| Watermill CQRS | Marshaler struct↔bytes + 事件名 | 双端共享 struct | 事件名版本化 v1/v2 |
| Temporal SDK | DataConverter + 函数签名反射 | 共享参数类型 | GetVersion 历史兼容 |
| CloudEvents SDK | SetData/DataAs + content-type codec | dataschema URI | spec 向后兼容原则 |

**建议：** event payload struct 放入 `contracts/event/{domain}/{event}/v1/payload.go`，双端共享，配合事件名版本化演进。

## 8. 审查分歧处理

### 分歧：L1 严重度（安全 P1 vs 架构 P0）

- **安全席位**判断：handler.go 内联有 `auth.FromContext + AnyRole("admin")` 检查，理论上 JWT middleware 注入 Principal 后仍有保护，降级为 P1
- **架构席位**判断：`auditQueryPolicy` 在生产注册路径从未被调用，Policy 挂载失败等同裸注册，P0
- **结论（合并）：** 采用架构席位判断——**P0**。原因：handler 内联检查依赖 JWT middleware 先执行（implicit deps），而 audit GET 路由如果被某个中间层 short-circuit 绕过 JWT，内联检查也会失效。正确做法是通过 `auth.Declare` 在路由层明确绑定 Policy，这是声明式安全的基础。
