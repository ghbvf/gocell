### F-1 [Cx1] auditcore.Init 未抽 `resolveEmitter` 子函数，与 accesscore/configcore 不一致

**文件**: `cells/auditcore/cell.go:152-219`（Init 内联 40+ 行 emitter 解析逻辑）
**对照**: `cells/accesscore/cell_init.go:31-82`、`cells/configcore/cell_init.go:74-124` 均已抽成 `resolveEmitter(mode)` 方法

PR 声称 F3 "三 cell 统一"，但 auditcore 仍把 hasEmitter/hasPending 分支检查、ResolveEmitter 调用、L2 warn、pending 清零全部展开在 `Init()` 里，accesscore/configcore 已抽成 `resolveEmitter`。功能等价，但：
1. 可维护性：未来改三条线时 auditcore 需要在 40 行 Init 里找；
2. 复杂度：Init 现在单函数 ~70 行，目测认知复杂度接近 15 上限，若再加一个字段容易触线；
3. 规范要求（CLAUDE.md）：函数认知复杂度 ≤ 15。

**建议**: auditcore 跟进同名 `resolveEmitter(mode cell.DurabilityMode) error`，保持三 cell 结构对称。/fix 一刀即可。

### F-2 [Cx1] auditcore.Init 里 `hasEmitter` 分支计算 `outcomeDurable` 两次

**文件**: `cells/auditcore/cell.go:168-175`

```go
var outcomeDurable bool
if hasEmitter && deps.DurabilityMode == cell.DurabilityDurable && !outbox.ReportDurable(c.emitter) {
    return errcode.New(...)
}
if hasEmitter {
    outcomeDurable = outbox.ReportDurable(c.emitter)  // 第二次调用
}
```

`ReportDurable(c.emitter)` 在 durable 分支被调用了 2 次（一次 guard、一次赋值）。accesscore 的 `resolveEmitter` 里只调一次（line 41-45）。

**建议**: 把两个 if 合并成 accesscore 那种写法：先 `outcomeDurable = outbox.ReportDurable(c.emitter)` 再做 durable guard。配合 F-1 一起修。

### F-3 [Cx1] `WithOutboxDeps(nil, nil)` 静默 no-op，调用方易写错

**文件**: `cells/accesscore/cell.go:131-143`、`cells/auditcore/cell.go:110-119`、`cells/configcore/cell.go:116-127`

godoc 已经写明 "Does NOT clear previously-set deps: `WithOutboxDeps(nil, nil)` is a no-op, not a reset"（三 cell 一致），但实现是 `if pub != nil` / `if writer != nil` 两个独立守卫，没有任何 log/err 提示。如果 caller 误以为 `WithOutboxDeps(nil, nil)` 可以重置（或者测试里 `var pub outbox.Publisher; WithOutboxDeps(pub, nil)` 把零值当成"取消上一次设置"），不会看到任何反馈。

**建议**（可选，**[Suggestion]**）: 保持语义，但在两个参数都是 nil 时记一条 Debug 或直接 panic（这是 Option pattern 的内部契约，panic 比静默吞更符合 fail-fast）。若保留当前语义，godoc 够清晰，不强求。

### F-5 [Cx1] `runtime/outbox/envelope_test.go` 删除丢失断言覆盖

**文件**: 已删除 `runtime/outbox/envelope_test.go`（394 行测试）

抽查 kernel 侧的测试文件（此 PR 未修改）应包含以下断言，否则属于覆盖回归：

- broken JSON 返回 `ErrEnvelopeSchema` 但不是 `ErrUnknownEnvelopeVersion`

**建议**: 走 `rg "schemaVersion.*schemaVersion\"|PascalCase" kernel/outbox/` 确认；若缺失，把对应 case 从删除的 runtime 测试搬到 kernel/outbox/envelope_test.go。非 blocking，但属于 Cx1 收口。

### F-6 [Cx2] archtest 扫描口子留有 "同名内部 import" 漏洞（悬而未解的 name-shadowing 问题）

**文件**: `tools/archtest/outbox_cell_test.go:74-108`（`isCellFile` 只匹配 `cells/<name>/cell.go`）

守卫规则：扫 `cells/<name>/cell.go` 发现顶层 `WithPublisher` / `WithOutboxWriter` 导出函数即 fail。但：

1. **多文件旁路**：`cell_extra.go`、`cell_options.go` 这类 sibling 文件被 `isCellFile` 严格排除（line 102-104 要求 `parts[2] == "cell.go"`）。如果以后重构把 Option 搬到 `options.go`，archtest 就失效了。
2. **`internal/` skip**：line 71 `filepath.SkipDir` 跳过 `internal`，合理；但 `cells/<name>/option.go` 或 `cells/<name>/cell_options.go` 不在 skip 列表也不在匹配列表，规则不覆盖。

当前 cells 都把 Option 放在 `cell.go`，不会触发，但这是"靠约定"不是"靠守卫"。

**建议**（**[Suggestion]**）: `isCellFile` 改成匹配 `cells/<name>/*.go`（排除 `*_test.go` + `internal/` + `slices/`），扫整个 cell 包顶层 decl。或者加一条 `TODO` 注释说明已知覆盖边界。非 blocking，本 PR 合入后 backlog 登记即可。
