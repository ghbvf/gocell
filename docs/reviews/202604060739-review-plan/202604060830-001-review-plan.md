# GoCell 全量代码 Review 执行计划 v3

## Context

GoCell 全量代码审查，覆盖全部 33 个 PR。

**核心策略**: 逐层 Review→Fix 交错执行 → 数据流合并审查 → PR 追溯 → 裁决收敛

**vs v2 关键改进**: 每层审完后立即修 P0，上层审查已修正的底层代码，防止级联假阳性。

---

## 设计原理

### 问题: 底层 P0 导致上层审查雪崩

如果 `kernel/outbox` 接口有缺陷，那么 `adapters/postgres`、`adapters/rabbitmq`、`cells/*` 的审查都会报告同一根因的不同症状。这些不是独立 findings，而是级联效应。

### 解法: 逐层 Review→Fix

```
Review pkg → Fix pkg P0 → Review kernel → Fix kernel P0 → Review runtime → Fix runtime P0
→ Review adapters → Fix adapters P0 → Review cells → Fix cells P0 → Review delivery
→ 数据流合并 → Fix 跨模块 P0 → PR 追溯 → 裁决
```

### P0 定义（Fix 判定标准）

以下问题立即修，不留 Fix Pack:

| 类别 | 示例 | 原因 |
|------|------|------|
| 安全漏洞 | SQL injection、算法混淆、credential 泄漏 | 外部可利用 |
| 数据丢失/损坏 | outbox 漏发、ACK 后数据未持久化 | 不可逆 |
| 语义错误 | 接口契约违反、一致性等级声明 vs 实现不符 | 上层依赖崩塌 |
| **外部输入未校验** | 用户输入未 sanitize、path traversal、XSS | 系统边界防线失守 |
| **外部输出泄漏** | 500 响应暴露内部细节、日志 dump 请求 body、error 含敏感信息 | 信息泄露 |
| **外部接口语义错误** | API 返回错误 status code、响应格式不符 `{"data":...}`、event payload 不兼容 | 消费方无法正确处理 |

**P1/P2 不在当层修**，记录到 findings 台账，留给最终 Fix Pack。

### Fix Wavefront: 底层修复导致上层编译/语义变化的处理

**问题场景**: 修了 `kernel/outbox.Writer.Write()` 接口（如加 `tx` 参数），`adapters/postgres` 和 `cells/*` 的实现编译不过。

**解法: 区分机械传播 vs 语义传播**

```
Fix kernel P0
  │
  ├─ 机械传播 (Mechanical Propagation)
  │   编译驱动的变更: 函数签名、import 路径、类型名称
  │   → 立即修，让 go build 通过
  │   → 不需要 review，因为变更是编译器强制的
  │   → 记录到 Fix Wavefront Log
  │
  └─ 语义传播 (Semantic Propagation)
      逻辑正确性的变更: "调用方传的值对不对？"
      → 不在当层修
      → 标记为 "待上层审查确认"
      → 上层 reviewer 带着这个标记做 review
```

**具体流程**:

1. Fix kernel P0（修改接口/逻辑）
2. 跑 `go build ./...` — 收集所有编译失败点
3. **机械传播**: 逐个修编译错误（改签名、改调用、改 import）
4. 跑 `go build ./...` — 确认编译通过
5. 跑 `go test ./kernel/...` — 验证 kernel 层测试通过
6. **记录 Wavefront Log**:
   ```
   Fix: R1B-Fix-01 kernel/outbox.Writer.Write() 添加 tx 参数
   机械传播:
   - adapters/postgres/outbox_writer.go:42 — 签名更新
   - cells/access-core/slices/session-login/service.go:78 — 调用更新
   语义待确认:
   - adapters/postgres/outbox_writer.go — tx 是否从正确的事务上下文获取？
   - cells/access-core — 调用方是否在事务内？
   ```
7. 上层 reviewer (R1D, R1E) 收到 Wavefront Log，**优先验证语义传播点**

**好处**:
- 全程 `go build` 通过 — 不会出现半修状态
- 上层 reviewer 知道哪些代码被底层 fix 改过，聚焦验证
- 机械传播不浪费 reviewer 时间
- 语义传播不被跳过（明确标记为 review checkpoint）

**边界情况处理**:
- 如果一个底层 fix 的机械传播影响 >10 个文件 → 暂停，先评估是否 fix 方案过于激进，考虑替代方案
- 如果底层 fix 导致上层测试失败 → 只修编译，不修测试失败（测试失败可能暴露上层自身问题，留给上层 reviewer）
- 如果底层 fix 方案有多种选择 → 选传播面最小的

## 执行拓扑

```
R0 (4)
  ↓
R1A: pkg (1) → Fix → R1B: kernel (5) → Fix → R1C: runtime (5) → Fix
  → R1D: adapters (6) → Fix → R1E: cells (5) → Fix → R1F+G (4) → Fix
  ↓
R2: 数据流合并 (6) → Fix
  ↓
R3A: Phase 1-2 PR (4) → R3B: Phase 3 PR (5)
  ↓
R4: 集成+覆盖 (5)
  ↓
R5: 裁决+Fix Pack (5)
```

**每层 Fix 原则**:
1. 只修 P0（安全漏洞、数据丢失、语义错误、**外部 I/O 边界问题**）
2. P1/P2 记录不修，留 Fix Pack
3. 修完跑 `go build ./...` + `go test` 验证
4. **Fix Wavefront**: 底层 fix 导致上层编译失败时:
   - 机械传播（签名/import）→ 立即修，记录 Wavefront Log
   - 语义传播（逻辑正确性）→ 不修，标记为上层 reviewer checkpoint
5. 上层 reviewer 带着 Wavefront Log 优先验证语义传播点
6. 如果某层 P0=0，Fix 阶段自动跳过
7. 如果机械传播 >10 文件 → 暂停评估 fix 方案是否过激
