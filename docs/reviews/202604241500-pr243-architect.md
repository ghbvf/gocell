## Findings

### Cx2-1 `signalsToWatch` 在 `shutdown.go` 未声明 → 文件未编译时 `Wait/NotifyContext` 找不到符号

`runtime/shutdown/shutdown.go:57,78` 引用 `signalsToWatch()`，但
`shutdown.go` 本身没有 build tag，而三份 `signals_{unix,windows,other}.go`
的 build constraints 必须完整覆盖所有 `GOOS`。当前：

- `signals_unix.go` → `//go:build unix`
- `signals_windows.go` → `//go:build windows`
- `signals_other.go` → `//go:build !unix && !windows`

这一组覆盖是完整的（unix∪windows∪其它 = all）。结论是 OK，但推荐增加
一条编译期守卫测试（`var _ = signalsToWatch`）以防止未来有人加第四个
`_foo.go` 改掉约束时出现某个 GOOS 下无 symbol 的死角。Cx2 而非 Cx1，因为
当前约束是互斥完整的。

建议：`runtime/shutdown/signals_smoke_test.go` 加一条 `TestSignalsToWatch_NonEmpty(t)`
跑在所有 GOOS，CI 的 os-smoke matrix 已覆盖 macOS+Windows，Linux 走主
build-test job。

### Cx2-2 `bootstrap.go` / `lifecycle.go` 仍带 `//go:build unix || windows` — unsupported 平台下 accesscore 不可编译

`cells/accesscore/initialadmin/bootstrap.go:1`、`lifecycle.go:1` 保留了
`//go:build unix || windows`，但 `path_default_unsupported.go` 声明的
`defaultStateDir` 会返回错误给调用方，意图是“其它平台也能 compile，
运行时 fail-fast”。bootstrap.go 带 build tag 意味着在 `GOOS=plan9` 等
平台下整个 `initialadmin` 包编译为空，`accesscore.WithInitialAdminBootstrap()`
会因符号缺失崩溃。

两种做法二选一：
1. 去掉 bootstrap.go / lifecycle.go 的 build tag，让 `platformImpl==nil`
   在 unsupported 平台的 ensureAdmin 路径上返回 panic/errcode（与
   path_default_unsupported 一致的 fail-fast 语义）。
2. 保留 build tag，但在 cell façade (`cells/accesscore/*.go`) 加 stub
   文件提供 unsupported 实现，保证 `go build ./...` 在任何 GOOS 通过。

当前 `os-smoke` 只测 macOS+Windows，所以这条不会被 CI 抓到；但 CLAUDE.md
的分层规则要求 `go build ./...` 在 kernel 以外层也通过。Cx2。

### Cx3-1 `platformCredfile` 接口命名与语义建议小修

`credfile_io.go:18` 的接口 `platformCredfile` 有三个方法
`EnsureSecureDir / SecureNewFile / VerifyOwnership`。语义分属两类：
前两个是“写路径上的安全原语”，第三个是“读路径上的完整性验证”。
分两个接口更符合 Kratos `transport.Server` / k8s `SecretProvider` 的
单职责切分。非阻塞。

### Cx3-2 `NotifyContext` 与 `Manager.Wait` 信号订阅重复注册风险

`runtime/shutdown/shutdown.go:55-67` `Wait()` 与 `NotifyContext()`
（第 77 行）都会调用 `signal.Notify/NotifyContext(signalsToWatch()...)`。
如果上层既用 `shutdown.NotifyContext(ctx)`（cmd/corebundle/main.go:38）
又在同一进程构造 `shutdown.New().Wait()`，两个订阅者都会收到信号 —
Go 标准库允许这样，但 Manager.Wait 的语义会被 NotifyContext 的 ctx
cancel 抢跑。

当前 `cmd/corebundle/main.go` 只用 NotifyContext 路径，`Manager.Wait` 没
有调用点（只有 `bootstrap.Lifecycle` 内部跑 hooks）。只是未来加使用者
时需要注意。建议在 `shutdown.go` 的 package doc 或 `NotifyContext` 注释
里加一句 “use one of NotifyContext OR Manager.Wait, not both in the
same process”。Cx3。

### Cx3-3 `path_default_*.go` 没 build tag 防御 — 文件名依赖 Go toolchain

`path_default_linux.go` 等文件是通过文件名后缀 `_linux.go` 让 go 工具链
隐式加 build constraint。这依赖于文件名约定而不是显式 `//go:build` 行
— 但四份文件都同时有显式 `//go:build linux` 等 build tag（已检查
linux / darwin / windows / unsupported 四份都写了显式行），所以覆盖
是 belt-and-suspenders。良好，不改。Cx3 只记一条提醒：新增 GOOS 支持
时必须同步更新 `path_default_unsupported.go` 的 `!linux && !darwin &&
!windows` constraint，否则会出现双重定义编译错误。可以考虑在
`docs/ops/env-vars.md` 加一条 state-dir 平台矩阵。

## 对标框架评估

- **Kratos transport / signal handling**: Kratos `app.go` 也用
  `signal.NotifyContext(ctx, syscall.SIGINT, syscall.SIGTERM)`。本 PR 的
  `NotifyContext` 签名等价但用 per-OS signal set 修正了 Windows 下
  SIGTERM 无法投递的隐患 — 这是对 Kratos 原实现的正确偏离，且注释
  (shutdown.go:73-76) 明确说明了偏离理由。采纳+偏离理由清晰。
- **k8s client-go filesystem/permissions_windows**: k8s `pkg/util/filesystem`
  对 Windows 走 DACL、Unix 走 mode bit 的切分模式与本 PR
  `credfile_security_{unix,windows}.go` 一致；PROTECTED_DACL +
  current user/Admins/LocalSystem 三 ACE 的选择与 k8s
  `pkg/kubelet/volumemanager/reconciler` 在 Windows 上的 secret 挂载
  ACL 策略相符。采纳。
- **systemd RuntimeDirectory**: `path_default_linux.go` 选择 `/run/gocell`
  是 systemd 约定；macOS 选 `$HOME/Library/Application Support/gocell/run`
  是 Apple File System Programming Guide 推荐位置；Windows 选
  `%LOCALAPPDATA%\gocell\run` 是 Microsoft Known Folders 推荐路径。
  三平台选择都有明确 reference，采纳。
