### C2 — `maybeMacOSHint` 与文档叙述存在逻辑冲突（应修复）

**文件:** `docs/operations/first-run-setup.md:246`（故障排查表，第 5 行）
**代码:** `cells/accesscore/initialadmin/bootstrap.go:228`

故障排查表中保留了一行：
```
启动失败 "credential dir not writable" | ... | macOS 请 `export GOCELL_STATE_DIR=$TMPDIR/gocell`
```

但 `bootstrap.go:228` 中的 `maybeMacOSHint` 函数仅在 `credPath` 以 `/run/` 开头时触发该提示——即用户手动指定了旧的 Linux 路径才会遇到。PR 的核心功能就是让 macOS 默认使用 `~/Library/Application Support/gocell/run/`，正常情况下不会再走 `/run/` 路径。

故障排查表中这行的"原因"列说"凭据目录无写权限或路径非法"，但"处理"列仍指向设置 `$TMPDIR/gocell`，而 macOS 自动路径节已经告知用户默认路径是 `~/Library/...`。这会使读者困惑：默认路径已经自动选好，为什么还要手动 export？

建议将处理列改为：`检查 $HOME 是否可读、~/Library/Application Support/ 是否有写权限；如需回退，export GOCELL_STATE_DIR=$TMPDIR/gocell`，与 macOS 启动报错写文件失败那行合并或保持一致的措辞（后者已在本 PR 中更新为"用户主目录路径解析失败（罕见）"，二者仍不对称）。

### C3 — `integration-testing.md` CI 命令与实际 workflow 触发条件描述偏差（信息性）

**文件:** `docs/guides/integration-testing.md:99`

文档写道：

> **When it runs:** on every PR (`pr-check.yml`) and every push to `develop` (`ci.yml`), in parallel with the Linux `build-test` job.

经核对 `.github/workflows/pr-check.yml` 和 `ci.yml`，两个 workflow 确实都传入了 `run-os-smoke: true`，描述基本准确。但文档的命令示例：

```
go test -count=1 -timeout 5m \
  ./runtime/shutdown/... \
  ./cells/accesscore/initialadmin/... \
  ./runtime/config/... \
  ./kernel/governance/...
```

与 `_build-lint.yml:180` 的实际命令完全一致（字面量匹配），可直接粘贴本地运行，符合可执行性要求。

唯一的小偏差：文档说 "in parallel with the Linux `build-test` job"——实际上 `os-smoke` 和 `build-test` 都在同一 workflow_call 下，并发度由 GitHub Actions 决定，并非文档中所说的强保证并行。这是措辞模糊，但不影响使用，列为 Informational。
