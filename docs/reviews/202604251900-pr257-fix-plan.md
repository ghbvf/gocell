## 二、修复批次与执行顺序

### Batch 1：典型 Cx2 — 类型 + 校验补强（4 项）

按依赖顺序：

1. **A1（DoS 反序列化保护）** — `kernel/outbox/observability.go` + `adapters/postgres/outbox_store.go`
   - 在 `ObservabilityMetadata` 加 `Validate()`：每字段 ≤ `idutil.MaxMetadataIDLen`（256B），4 字段总和 ≤ 1KB
   - `scanClaimedEntry`：`if len(observabilityJSON) > maxObservabilityJSONBytes { Warn + skip }`，用 `const maxObservabilityJSONBytes = 8 << 10` (8KB)
   - 写入端 `InjectObservabilityFromContext` 后调用 `Validate()`，超长字段 zero 化（防御写入）
   - 测试：1KB+ 注入被截断；8KB+ JSONB 被忽略

2. **A2（Entry.Validate 拒绝保留键）** — `kernel/outbox/outbox.go::Entry.Validate`
   - 加 `var reservedMetadataKeys = []string{"trace_id", "traceparent", "request_id", "correlation_id", "span_id", "trace_state", "tracestate"}`
   - 在 `validateMetadata` 头部检查；命中即 `errcode.ErrValidationFailed` "metadata key %q reserved, use Entry.Observability"
   - 测试：3 case（命中 trace_id / 命中 traceparent / 全部不命中）

3. **A3（integration NULL 路径）** — `tests/integration/outbox_fullchain_test.go`
   - 新 sub-test `WriteEntryWithZeroObservability_ReadsBackZero`
   - 写一个 `Entry{Observability: ObservabilityMetadata{}}`（zero），验证：DB 列 IS NULL；consumer 读出 zero struct；不输出 unmarshal warn

4. **A6（middleware presence 检测）** — `kernel/outbox/observability.go`
   - 加 `RegisterObservabilityRequirement(*Subscriber, sub Subscription)` helper（或在 ConsumerBase setup 阶段 self-check）

   - 实施：在 `ObservabilityContextMiddleware` godoc 加 "MUST be installed when consuming entries written by `InjectObservabilityFromContext`; bootstrap.WithDefaultMiddleware composes this automatically"
