# v3 ↔ GoCell 对照索引

> 本文是《Vibe Coding 工程化治理 v3》方法论（`~/Documents/methodology/v3.md`）与 GoCell 项目的对照索引。**v3 是 1 周前产出的探索性理论，未经工程实践验证，不是真相源**。本文档的作用是"用 v3 提问，让 GoCell 实践来回答"——实践证据 > v3 理论。

## 关键认知前提（重要）

### v3 在认识论上是什么

- **v3 是探索性理论**：2026 年 5 月 1 日产出，~8000 字，七大思想融合，**没有任何项目实践验证过**
- **v3 是 advisory（参考），不是 normative（规范）**：详见 [三层关系 ADR](202605020052-adr-methodology-three-layer.md)
- **v3 可能错**：其中某些主张可能在 GoCell 实践中被证伪、被抛弃、被局部改写

### 实践与理论的正确关系

```
错误关系（前一版本采用）：     正确关系（本版本采用）：
                                
   v3.md                        GoCell 实践（PR / test / bug）
     │                                │
     ↓ "真相源"                       ↑ 反向校准
                                      │
   GoCell PR ←抄                  v3.md（被质疑、被修订）
```

具体含义：
- 实践证据 > v3 理论。v3 与 GoCell PR review 冲突时，PR review 优先
- 实践有权**质疑、修改、抛弃** v3 中不工作的部分
- v3 的价值在"提供推理框架与提问视角"，不在"提供答案"
- **GoCell 自己的 PR / test 结果 / bug 报告才是真相**

## v3 主张中需要被 GoCell 实践验证或可能被证伪的清单

下列 v3 主张是"探索性的、可能错的"，**GoCell 实践有权在落地中推翻它们**。每个主张应有"survive / modified / refuted"三态跟踪。

| 主张 | v3 节 | 可能的证伪场景 | 验证窗口 |
|---|---|---|---|
| 五态机适用于所有模块 | 第 8 部分 | GoCell 中存在天生不适合状态机的 cell（纯库 / utilities） | GD1 |
| 三维健康度的 60/60/60 阈值 | 第 8.5 节 | 阈值过高，导致绝大多数 cell 无法通过 | GD2 末 |
| 价值/结构/工程权重 1/3 各占 | 第 8.5 节 | 实际权重应按 cell 性质动态调整 | GD2 末 |
| AI 接受率 70-90% 是合理区间 | 第 9 部分阶段二 | 单人项目可能 50% 就够，或 95% 仍合理 | GD4 末 |
| 利特尔法则 WIP = 周期 × 吞吐量 直接套用 | 第 5.4 节 | 单人项目无 WIP 概念，应改为时间块策略 | GD3 末 |
| reconciliation 30 min cron 频率 | 第 6.5 节 | GoCell 的特定 drift 周期可能要求更长或更短 | GD5 末 |
| MDM 声明式范式必然适合代码治理 | 第 6 部分 | 代码治理的"漂移"语义不同于设备 desired state | GD5 |
| Harness 五件套全要 | 第 6.6 节 | GoCell 单人项目可能不需要"评估循环"作为独立组件 | GD4 |
| 资本论分析框架对 AI 时代仍适用 | 第 3 部分 | 类比可能局限，AI 不只是劳动力倍增器 | GD2 |
| 决策审计员必须独立角色 | 第 10 部分 | 单人项目可能 .claude/agents 配置就够 | GD6 |

**预期结果**：GD1-GD6 完成后，上表中应有若干条标 "modified" 或 "refuted"，反向回写到 v3.1。**如果 11 条全部 "survive" 反而可疑**——说明实施可能没有真正质疑 v3。

## 反向：GoCell 反过来给 v3 的启发

GoCell 实施过程中发现的 v3 不足，应**从 GD1 开始**（不是等到 GD4）持续回写到 v3.1：

- 早期已知可能的偏离（待验证）：
  - v3 假定能从 0 锻造框架，但 GoCell 显示"框架自然演进 + 反复重构"才是常态
  - v3 假定多人协作，单人项目特殊性应进入 v3.1 章节
  - v3 假定每个栈实施成本相近，实际 .NET / Java 后端的 GD5 可能比 Go 简单

- 回写机制：
  - 每个 GD 末尾自动开 `v3-feedback` issue，记录本 GD 中发现的 v3 偏离
  - GD6 季度评审时汇总，触发 v3.1 修订
  - **不等到 GD6**——若任一 GD 发现 v3 主张被证伪，立即在本文件标 "refuted"
