# Wave 4 触发型提前推进计划（3 条临界条目）

**生成日期**：2026-05-12
**关系**：本计划是 [`docs/plans/202605112000-036-archtest-governance-rollout-plan.md`](202605112000-036-archtest-governance-rollout-plan.md) §4 Wave 4 触发型清单的**分流方案**。036 保持不动作为决策原则视角；本文件挑出**触发距离 = 1** 的 3 条条目（再有一个事件即触发），按 ROI 排期主动收口，避免被动响应延迟。

**核心论点**：当 trigger 条件距离 = 1（即"下一个就触发"）时，**主动做 ROI 普遍优于等触发**，原因：
1. **反向触发（事故首现）= 已晚一步**：事故已发生，治理上线只能阻止下一次
2. **被动 retrofit 成本翻倍**：trigger 后既有新增量已铺开，需先 ship #N + 1 → 再回头改 funnel + 旧 N 个，PR 边界混乱
3. **当前量级稳定 = 设计空间完整**：还可以从容设计 API、配 ADR、补 archtest；推后任一项都会被既有形态约束

**适用范围**：仅 charter §4 Wave 4 触发型清单。Wave 2 / Wave 3 已有 plan（036 §3）覆盖。

### 1.1 PR-A1：FINDFIRSTCHILD-TYPED-API-01 ✅ **已完成**

> **Ship 记录**：`scanner.FindFirstChild[S any, N interface{*S; ast.Node}](root, predicate func(N) bool) (N, bool)`（codebase-一致签名，复用 EachInChildren，单一内部 sentinel）。实测 **9 处** EachInChildren find-first sentinel 全迁移（计划估 6+；含 scanner_framework_usage_test.go 自身被新规则 dogfood 抓出的第 9 处）。`SCANNER-FRAMEWORK-USAGE-02`（canonical ID；描述性 "closure-done-sentinel ban" 入 prose，原拟长 ID 违反 INVENTORY-ANCHOR-VALID-ID-01 grammar）allowlist=0 + RED/GREEN fixture + 盲区(BS1/BS2)反向自检。**范围决策**：仅 depth-1；EachInSubtree sentinel 家族（递归 find-first / 子树存在性断言，正交轴）拆出独立触发型 backlog `FINDFIRSTINSUBTREE-API-01`（cap-14 §14.1）。AI-rebust：下游 Hard（N 选错=编译失败）+ 上游 Medium（archtest 形态 ban，已是 Go 天花板，无 phantom 升级路径，详见 USAGE-02 godoc）。

**目标**：把 6+ 处 `closure + done bool` sentinel helper 一锅迁移到 typed `scanner.FindFirstChild[N]` API + archtest 拦再生，**Hard 化 walk API 第三轴**。

**Walk API 三轴现状**（charter Wave 2 已 Hard 两轴）：
| 轴 | API | AI-rebust | 状态 |
|---|---|---|---|
| 节点类型 | `scanner.EachInSubtree[N]` / `EachInChildren[N]` typed callback | **Hard**（编译期不可表达类型错误） | ✅ PR #460 |
| 深度选择 | `EachInSubtree` (recursive) vs `EachInChildren` (depth=1) | **Hard**（fixture-level + compile-level 双保障） | ✅ PR #460 |
| **第三轴：closure+done 模式** | 当前 `var found bool; EachInChildren(... if cond { found = true; done = true })` 6+ 处 | **Soft**（手写 sentinel + done flag 习惯） | 🔴 本 PR |

**当前 6+ 处 helper sites（Wave 3 Phase 3.10 完成后 grep）**：
- `tools/archtest/outbox_invariants_test.go:420` — `found bool` 形态（HandleResult 字段扫描）
- `tools/archtest/outbox_invariants_test.go:1242` — `done = true` 形态
- `tools/archtest/outbox_invariants_test.go:1382` — `done = true` 形态
- `tools/archtest/outbox_invariants_test.go:1612` — `found bool` 形态
- `tools/archtest/security_defaults_test.go:861` — `hasKey(cl, "Authenticator")` typed helper（已 typed 但与 FindFirstChild 形态等价）
- `tools/archtest/contract_spec_*` ×2（charter §4 cap-14 PR460-FU 列出）

**修复方案**：
1. **framework**: 新建 `tools/archtest/internal/scanner/findfirstchild.go`
   ```go
   // FindFirstChild walks node's direct children, returns first child of type *N
   // matching predicate, or zero value + false if none found.
   // Stops on first match (no manual done flag needed).
   func FindFirstChild[N ast.Node](node ast.Node, predicate func(*N) bool) (*N, bool)
   ```
   类型签名: 节点类型 N 编译期固定（Hard 轴 1）+ 单一 children 遍历（Hard 轴 2）+ 隐式 early return（Hard 轴 3，没有暴露的 done flag）
2. **bulk migration**: 6 处 sentinel helper → `FindFirstChild[N]` 一锅替换
3. **archtest enforcement**: 在 `tools/archtest/scanner_framework_usage_test.go` 增加新规则 `SCANNER-FRAMEWORK-USAGE-02-CLOSURE-DONE-SENTINEL-BAN`（命名待定），扫禁 `var done bool` + `EachInChildren` / `EachInSubtree` callback 内 `done = true` 模式；allowlist = 0（迁移已 100%）

**AI-rebust 升级**：**Soft (closure + sentinel 手写习惯) → Hard (typed funnel + archtest 拦)**

**工时**：3-4h dev / 1h review（charter §4 原估，charter 标注"6 处 sentinel helper 总改动 < 200 LOC，review 面积可控"）

**为什么不等触发**：
- charter §4 原 trigger："第 7 处 closure+done sentinel helper 出现"
- 当前 6+ 处已经在 charter estimate 范围（事实上等于 charter 上限）
- 等 #7 出现意味着先 ship #7 → 再回头改 7 处 + 立 archtest，PR 边界混乱
- 提前做：当前 6 处 + 0 新增 = 一锅迁移；ROI 至少高 1.5x

**依赖**：Phase 3.10 (CELLGEN-LITERAL-FUNNEL-02) 可并行（不同子目录）；Phase 3.9 PR-S7 可并行

**范围限制**：
- ✅ scanner.FindFirstChild[N] API + 6 处迁移 + archtest enforcement
- ❌ 不做 FindFirst 跨深度（subtree 版）——单独触发型，需要时再加 `FindFirstInSubtree[N]`
- ❌ 不做 `FindAll` 形态——已被 EachInChildren cover

**验证**：
1. `scanner.FindFirstChild[N]` 编译通过 + table-driven 单测覆盖 ≥ 90%
2. 6 处迁移后 `go test ./tools/archtest/...` GREEN
3. 新增 archtest 规则 RED fixture（手写 `done = true` 形态）+ GREEN fixture（FindFirstChild 用法）

### 1.2 PR-A2：ARCHTEST-CARVEOUT-NARROW-FUNCLEVEL + B2-K-08-CARVEOUT-NARROW（合并 PR）⭐⭐ **强推荐**

**注**：`B2-K-08-CARVEOUT-NARROW` 与 `ARCHTEST-CARVEOUT-NARROW-FUNCLEVEL` 是同根条目（前者 PR#391 K#08 原始 follow-up；后者 PR#391 F2 加 ADR 强制）；本 PR 同时关闭。**不要**与 `B2-K-08`（Assembly race test 认知复杂度，cap-14-tooling.md line 22 不同条目）混淆。

**目标**：把 `errcode_invariants_test.go` 现有 2 处 file-level carve-out（`pkg/ctxcancel` + `pkg/httputil`）改造为 **function-level carve-out**（仅豁免具名 helper 函数体内 struct literal），同时立 ADR 登记 carve-out 清单 + 理由，防 carve-out 失控蔓延。

**现状漏洞**：
- `errcode_invariants_test.go:64-65` 把 `pkg/ctxcancel` + `pkg/httputil` 整包豁免（`MESSAGE-CONST-LITERAL-01` / `ERRCODE-KIND-LITERAL-01`）
- 真实需要豁免的只有 2 个 function：`ctxcancel.WrapOrInfra` + `httputil.writeErrcodeError`（struct literal 内嵌 error，runtime fmt 必要）
- 整包豁免 = soft over-permission：包内其他文件/函数也走 struct literal 不会被守
- 第 3 处 carve-out 一旦出现，整包豁免模式继续扩 → 守卫退化

**修复方案**：
1. **framework**: archtest 改 carve-out 模型从 `packagePath` allowlist 改为 `(packagePath, functionName)` 二元组 allowlist
   - 新增 helper `func carveOutFn(pkg, fn string) carveOut`
   - `messageConstLiteralCarveOuts` / `errcodeKindLiteralCarveOuts` 改为 `[]carveOut`
2. **bulk migration**: 2 处现有 carve-out 改造为 function-level
   - `{pkg: ctxcancel, fn: "WrapOrInfra"}`
   - `{pkg: httputil, fn: "writeErrcodeError"}`
3. **ADR**: 新建 `docs/architecture/202605121800-adr-archtest-carveout-narrow.md` 登记 carve-out 清单 + 每条理由 + 升级路径（"若新增 carve-out 必须 ADR amend"）
4. **archtest 自检**：扫 carve-out 列表只能扩展，不能删除（防 silent retreat）

**AI-rebust 升级**：**Soft (整包 file-level 豁免) → Medium-偏-Hard (function-level + ADR 登记)**

**工时**：4-6h dev / 1h review
- archtest helper 改造：1.5h
- 2 处迁移：0.5h
- ADR 起草：1.5h
- 测试 + verify：0.5h

**为什么不等触发**：
- 原 trigger："第 3 个 file-level carve-out 出现"
- 等触发 = 被动追认；先 ship 第 3 个 file-level → 再回头 narrow 3 个，本质是 retrofit
- 主动做：2 → 3 临界点；既有 2 处 carve-out 形态完整可控；ADR 一次到位（不需要 "第 3 个 carve-out 时再加 ADR"）

**依赖**：无（与所有 Wave 3 phases 文件不重叠）；可与 PR-A1 / Phase 3.9 / 3.10 完全并行

**范围限制**：
- ✅ MESSAGE-CONST-LITERAL-01 + ERRCODE-KIND-LITERAL-01 两条规则的 carve-out narrow
- ❌ 不触其他 archtest 规则的 carve-out（如 OUTBOX-HANDLERESULT-FACTORY-PREFERRED-01 已经是 function-aware allowlist；不需要改）
- ❌ 不立 universal carve-out framework（首条 ADR 锁本 PR 范围，未来其他规则需要 carve-out narrow 时单独立 ADR amend）

**验证**：
1. 改 carve-out 后 `go test ./tools/archtest/...` GREEN
2. RED fixture：在 `pkg/ctxcancel/util.go`（非 `WrapOrInfra`）写 `errcode.Error{Message: fmt.Sprintf(...)}` → archtest 红
3. ADR `Date` 字段 = 文件名前缀日期（防 archtest ADR-DATE-CONSISTENCY-CHECK 后续做时漂移）

### 1.3 PR-A3：SERVICEOWNED-OWNERSHIP-GUARD-01 ⭐⭐ **中强推荐**

**目标**：把 `auth.serviceOwned` 对象级授权从"个例 review 把关"升级为 governance rule + archtest 守卫，**在 endpoint = 1 时设计完整契约**，避免 endpoint > 1 后 retrofit 风险。

**现状**：
- 当前唯一 serviceOwned endpoint：`contracts/http/auth/session/delete/v1/contract.yaml:22 serviceOwned: true`
- service 层负责 owner match 检查，但治理层无机器可验证的 ownership enforcement 钩子
- 既有 endpoint 形态尚未约束新增 endpoint 必须遵循的契约模板

**漏洞模型**：
- 第 2 个 serviceOwned endpoint 出现时，若 service 层 owner check 漏写 → 静默漏权
- 现有 endpoint 的 service 实现没有强制要求"owner 检查必须在 authorization phase 完成"
- contract.yaml `serviceOwned: true` 标记没有触发任何 governance / archtest 钩子

**修复方案**：
1. **contract 层契约定义**：
   - contract.yaml `serviceOwned: true` 时必须声明 `ownership` 字段（`subject_path` / `resource_path`）
   - 例：`session/delete` 的 ownership: `{subject: "ctx.subject_id", resource: "path.session_id.user_id"}`
2. **governance rule（kernel/governance）**：`OWNERSHIP-DECLARATION-REQUIRED-01`
   - validate.go 拒绝 `serviceOwned: true` 缺 `ownership` 块
   - validate.go 拒绝 `subject_path` / `resource_path` schema 不合法
3. **archtest L5**：`SERVICEOWNED-HANDLER-OWNER-CHECK-01`
   - 扫 cell adapter handler：声明 serviceOwned 的 endpoint，handler 必须显式调用 `auth.RequireOwner(ctx, subject, resource)` 或等价契约
4. **既有 endpoint 迁移**：`session/delete` 增加 `ownership` 块 + handler 改造（如果当前未显式调用）

**AI-rebust 升级**：**Soft (review 抓 owner match) → Medium-偏-Hard (governance rule + archtest L5 双层守)**

**工时**：6-10h dev / 2-3h review
- contract schema 扩展：1.5h
- governance rule 实现 + 单测：2-3h
- archtest L5 + RED/GREEN fixture：2-3h
- session/delete 迁移：1-2h
- 文档：1h

**为什么不等触发**：
- 原 trigger：`auth.serviceOwned` endpoint > 1
- endpoint 数 1 → 2 临界点；既有 1 个 endpoint 是孤例，treatment 完整设计空间打开
- 等 endpoint = 2 时，第 2 个 endpoint 已经按某种形态 ship（可能与 governance rule 假设不一致），retrofit 成本 = 改 2 处 endpoint + 立 governance + 立 archtest
- 主动做：1 处迁移完整契约 + governance + archtest 一次到位

**依赖**：无；可与 PR-A1 / PR-A2 / Phase 3.9 / 3.10 完全并行（不同文件子树）

**范围限制**：
- ✅ contract.yaml `ownership` 块 + validate.go 拒空 + archtest L5 + session/delete 迁移
- ❌ 不做 ownership 跨 cell 传播守（subject_id ↔ session.user_id 跨服务对账，留触发型）
- ❌ 不做 ownership 缓存层（auth-cache 已是另一条 backlog）

**验证**：
1. `gocell validate` 对一个故意缺 ownership 块的 `serviceOwned: true` contract → fail-fast
2. archtest L5 RED fixture：handler 缺 `auth.RequireOwner` → archtest 红
3. session/delete 既有 contract + handler 通过新 governance + archtest GREEN
4. 文档更新：`.claude/rules/gocell/auth.md` 或类似位置加 "serviceOwned 必须声明 ownership 块" 章节
