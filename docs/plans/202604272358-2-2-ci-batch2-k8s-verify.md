# L3 计划：K8s 风格 CI 治理 第二批（make verify 入口 + 5 条 hard gate，单 PR 彻底消化）

## Context

第一批 plan（`docs/plans/202604270020-1-2-ci-3-claude-ship-reactive-bachman.md`）已合并 PR-MODE-1~8 八条 archtest，PR #307 在收尾 PR-MODE-6.1（属第一批）。第一批仍有 Phase 2（Claude 分层骨架）/ Phase 3（v1 schema ADR）/ Phase 4（6 角色 retrospective）未启动。

本 plan 对标 Kubernetes `hack/verify-*` 治理体系，把散落的 archtest+governance+脚本统一到 `make verify`，并推进用户列出的 5 条优先 hard gate（GENERATED-DRIFT / TEST-NO-SLEEP / PANIC-REGISTERED / V1-RESPONSE-EVOLVE / SHELL-SAFETY）。其中 V1-RESPONSE-EVOLVE 把第一批 Phase 3 收掉。

**核心节奏**：保留第一批「1 模式 1 PR + 同 PR 业务消化 + CI 始终绿」**不引入 baseline / NEW-CLEAR-HARD 分批**。每条规则一次性彻底消化全仓存量（包含 87 处 `time.Sleep` 与 56 处 production panic），开 PR 前在 worktree 跑通 archtest 0 violations。

**勘察实测（2026-04-27 develop）**：
- `make verify` **不存在**；workflow `governance.yml` / `_build-lint.yml` 直跑 `gocell validate --strict` / `go test ./tools/...`
- `tools/archtest/` 18 文件（已含 LAYER-01~10、SEC-FAIL-CLOSED-01~05、ERROR-FIRST-API-01、META-QUERYPARAM-DRIFT、ADV-06）
- `kernel/governance/rules_strict_extra.go` 含 FMT-20（response strict）
- `gocell generate assembly --id=corebundle` 写 2 文件（`cmd/corebundle/main.go` + `assemblies/corebundle/generated/boundary.yaml`），均 git 跟踪，当前无漂移
- `time.Sleep` 实测 **87 调用 / 35 测试文件**（adapters/rabbitmq 30、runtime/* 18、其他 39）
- production panic **56 处**；ADR `docs/architecture/202604270030-architectural-panic-whitelist.md` 当前白名单 1 项；router.go(5)、distlock(5)、sessionlogin(5)、sessionrefresh(5)、memstore(3)，余 ~33 散落
- `contracts/http/*/v1/response.schema.json` **30 文件**，100% 顶层 `additionalProperties:false`
- `scripts/seed-test-data.sh` 缺 `set -euo pipefail`（其余 2 个脚本合规）

**对标 K8s（已勘明）**：
| 元素 | K8s 实现 | 本 plan 借鉴 |
|---|---|---|
| 入口 | `hack/make-rules/verify.sh` glob `hack/verify-*.sh` | `make verify` 自动发现，新增 gate 只增文件 |
| 隔离 | `git worktree add` 临时工作树 | GENERATED-DRIFT 实现路径 |
| 累积 | fail-accumulate（不 fail-fast） | 所有 verify-* 跑完再统一 exit |
| 测试质量 | `verify-test-code.sh` 直接 grep / 零容忍 | TEST-NO-SLEEP / PANIC archtest 直接零容忍 |

## 用户硬约束（彻底 / 不向后兼容 / 优雅简洁）

1. **每个 CI 模式 1 个 PR**，archtest + 同 PR 业务消化全仓存量，CI 始终绿
2. **不留** waiver / lenient / opt-in flag / `//nolint:archtest-XXX` / **baseline 文件锁存量** / **NEW-CLEAR-HARD 三阶段**——任何「先豁免后清零」都是向后兼容 shim
3. **不留** TODO / placeholder / 决策 follow-up——所有路径选择本 plan 内拍板
4. **不新建辅助包/迁移文档**当现有库够用——`testify.Eventually` 替代新写 `pkg/synctest`；不写 `no-sleep-migration.md`
5. **不保留旧规则名**——FMT-20 直接改语义，不拆「FMT-20-REQUEST + 删响应」造别名
6. 中文输出；路径/命令保留原样

### 阶段 0 — PR-CI-0 VERIFY-ENTRYPOINT-01（~4h，无依赖）

**抽象**：`make verify` 入口 + glob 自动发现，后续 gate 只增 `hack/verify-X.sh`。

**改动**：
- 新 `hack/make-rules/verify.sh`：
  ```bash
  set -euo pipefail
  cd "$(dirname "$0")/../.."
  fails=()
  for f in hack/verify-*.sh; do
    [[ -e $f ]] || continue
    bash "$f" || fails+=("$f")
  done
  [[ ${#fails[@]} -eq 0 ]] || { printf 'FAIL: %s\n' "${fails[@]}"; exit 1; }
  ```

- 改 `.github/workflows/_build-lint.yml`：现有 `go test ./tools/...` step 替换为 `make verify`（archtest 已被 `verify-archtest.sh` 覆盖）

**业务消化**：无（行为等价迁移）；governance.yml 和 _build-lint.yml 同步收敛**不保留旧 step 双路径**

## 决策点（已采用方案）

| 决策 | 已选方案 | 理由 |
|---|---|---|
| baseline 三阶段 vs 单 PR 一次消化 | **单 PR 一次消化** | 用户硬约束「不向后兼容 / 彻底」；baseline 文件本质是 backwards-compat shim |
| FMT-20 拆分 vs 改语义 | **改语义保留 ID** | 不留别名；语义改动在 ADR + test 中说明 |
| 响应顶层放宽 vs 顶层+nested 全放宽 | **全放宽** | 响应 schema 既然按方向 A 演进自由就该彻底；保留 nested 严格是半截兼容 |
| `pkg/synctest` 新包 vs 用现有库 | **`testify.Eventually` + stdlib** | 优雅简洁；不引入新抽象 |
| `no-sleep-migration.md` 文档 | **不写** | 直接体现在 diff；项目无外部消费方 |
| router.go 5 panic：refactor vs 永久豁免 | **永久 ADR 白名单** | mux 注册期 = init 语义；refactor 侵入过大违背简洁 |
| `update-generated.sh` 配对脚本 | **不建** | `make generate` 已存在，verify-generated 内部直接调，无需配对脚本 |
| Generated header 加 `// Code generated` | **不加** | 项目仅自身消费；README 注明 + verify 强制即足够 |
| `_build-lint.yml` 是否保留 archtest step | **替换为 `make verify`** | 不留双路径 |
| 6 角色 retrospective 并入本 plan | **不并入** | 应在 Claude 分层骨架（第一批 Phase 2）合并后再跑 |
