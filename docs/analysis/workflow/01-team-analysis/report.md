## 1. 分析背景与问题定义

### 1.1 触发原因

当前 8 阶段多角色工作流中，Harness 专家仅在**阶段 2（spec 审查）**出现一次，之后完全消失。但从 Phase 0-2 的历史教训表来看，绝大多数问题（跳步、角色缺席、流水线绕过）都是**阶段执行过程中的流程偏差**，而非 spec 质量问题。

### 1.2 分析目标

- 诊断 Harness 专家在当前工作流中的角色缺口
- 从 4 个角色视角提出改进方案
- 评估改进的可行性和成本

## 2. 架构师视角

### 2.1 现状诊断

#### Harness 专家的参与频率

| 角色 | 参与阶段 | 持续性 | 硬阻塞权 |
|------|---------|--------|---------|
| 总负责人 | 0-8 全程 | 持续 | 是 |
| PM | 2, 5.4, 8.3 | 每 batch | 是（8.3 PASS/FAIL） |
| 文档工程师 | 5, 8.2 | 两阶段 | 否 |
| 架构师 | 2, 6（升级） | 按需 | 是（6.10 P0） |
| **Harness 专家** | **2** | **一次性** | **否** |
| QA | 7 | 一次 | 是（qa-report 硬门） |

**核心发现**：Harness 专家与 Roadmap 规划师并列为参与频率最低的角色。但两者有本质区别——Roadmap 只需要一次性范围确认，而 Harness 专家的职责（SLE 内核集成正确性）是贯穿全过程的持续关注点。

#### 历史教训中 Harness 专家缺位的后果

Phase 1 的 5 个 P0 bug 中，至少 3 个是 Harness 专家在**回溯审查**时才发现的：

| Bug | 严重度 | Harness 编号 | 描述 |
|-----|--------|-------------|------|
| TD-P0-4 | 系统完全不工作 | Harness #22 | 请求 context 取消杀死后台 orchestrator |
| TD-P1-4 | 竞态条件 | Harness #20 | Cancel/Complete 并发写入 run 状态 |
| TD-P1-5 | 资源泄漏 | Harness #17 | 每次 run 创建新 ServiceGraph/DB 连接 |
| TD-P1-6 | 功能不完整 | Harness #18 | EvidenceBridge 只挂 AfterGate/AfterReviewFull |

**如果 Harness 专家在 Phase 1 阶段 2 正常参与，这些问题可以在编码前被发现。**

### 2.2 架构可持续性分析

#### 风险链 1: 设计-实现偏差累积（Design-Implementation Drift）

阶段 2 的设计建议经过 3→4→5 三步传递，每步都可能偏差。当前没有人专门检查"实现是否忠实于 Harness 专家的内核集成设计"。PM 在 5.4 只看 build/test 绿不绿，6 视角 Reviewer 看代码质量，但没人看"设计-实现对齐"。

#### 风险链 2: Tech Debt 在内核层累积

Phase 0→1→2 的内核级 tech debt 约占 60%。没有 Harness 专家参与收尾，修复优先级可能被低估。例如 TD-P1-5（每次 run 创建新 ServiceGraph）如果不修，Phase 3 多租户时连接池会被耗尽——这个因果链需要 Harness 专家来判断。

#### 风险链 3: 架构退化无人感知

6 视角 Reviewer 检查代码风格和安全性，但不检查"这个改动是否违反了 SLE 的架构约束"。

### 2.3 Harness 专家应扮演的角色

三种定位对比：

| 定位 | 描述 | 问题 |
|------|------|------|
| SRE | 保障运行时可靠性 | SRE 面向运行时，Harness 面向构建时 |
| 审计员 | 事后验证合规性 | 事后发现问题代价太高（Phase 1 的 5 个 P0） |
| **控制平面（推荐）** | 关键节点做准入/准出检查 | 类似 K8s Admission Controller |

**推荐控制平面定位的理由**：SLE 自身就是"验证管线"产品（gate→review→triage→fix loop），让 Harness 专家在工作流中扮演同样角色，是产品理念和项目管理的一致性体现。

### 2.4 架构师的 8 条改进建议

| # | 阶段 | 改进 | 新增 Agent | 关键路径增加 |
|---|------|------|-----------|------------|
| 1 | S2 | 输出结构化为 `harness-constraints.md` | 0（改 prompt） | 0 |
| 2 | S4 | tasks.md 补充审查 | +1 | ~5 min |
| 3 | S5.4 | PM 对照 harness-constraints.md 标记实现状态 | 0（PM 做） | ~2 min/batch |
| 4 | S6 | 6 视角 prompt 注入 harness-constraints.md | 0（改 prompt） | 0 |
| 5 | S7 | 内核集成烟雾测试（与 QA 并行） | +1 | 0（并行） |
| 6 | S8 | Tech Debt 趋势分析（与 8.2 并行） | +1 | 0（并行） |
| 7 | 每 2 Phase | 深度 Harness 架构审计 | +1 | ~20 min |
| 8 | S0 | 角色完整性检查增加"上次跳过记录" | 0 | ~1 min |

**总开销**：每个 Phase 增加约 3 个 Agent 调用。由于建议 5/6 都与现有阶段并行，实际关键路径增加约 5-10 分钟。

### 2.5 防止被跳过的四层防护

| 层级 | 机制 | 强度 |
|------|------|------|
| 层 1 | harness-constraints.md 为 S3 硬入口条件 | 最强 |
| 层 2 | 跳过时必须在 decisions.md 记录理由 + tech-debt.md 记录 PROC-DEBT | 中 |
| 层 3 | S6 增加 Harness 补偿 review 视角 | 事后补救 |
| 层 4 | 连续 2 Phase 跳过触发红色警告 | 断路器 |

## 3. PM 视角

### 3.1 交付效率分析

#### 问题发现时间分布

| 问题 | 产生于 | 发现于 | 延迟量 |
|------|--------|--------|--------|
| Harness 审查被跳过 | Phase 0 S0 | Phase 0 S2（用户提醒） | 2 阶段 |
| Harness 审查被跳过 | Phase 1 S0 | Phase 2 回溯 | 1 个 Phase |
| QA 被跳过 | Phase 1 S4 | Phase 2 期间 | 1 个 Phase |
| 文档+DevOps 缺失 | Phase 1 S4 | Phase 2 补做 | 1 个 Phase |
| QA 被跳过 | Phase 2 S4 | 合并后发现 | 合并后 |
| PM 跟踪缺失 | Phase 2 S5 | 用户提醒 | 跨阶段 |
| Speckit 被绕过 | Phase 0-2 S1/S4 | PM 审计 | 跨 3 Phase |

**10 个问题中 7 个在下一 Phase 或更晚才发现。** 工作流缺乏"当 Phase 内的流程合规性检查"。

#### 返工时间估算

- Phase 2 中 T010.1（修复 Phase 1 的 5 个 P0）：占 Sprint A 约 8% 工作量
- Phase 2.5（整个 905-phase25-backfill）：纯返工，约占正常 Phase 的 30-40%
- Phase 2 设计阶段：因 Phase 1 遗留问题至少多了 1 轮 spec 迭代

**预防性检查成本（每 Phase ~35 分钟）vs 返工成本（1.3-1.8 小时）= ROI 约 2.2-3.1 倍**

### 3.2 角色协作网络

当前 Harness 专家是**孤立节点**——只在 S2 与总负责人有一次信息交互，之后完全断开。

| 角色 | 与总负责人交互的阶段数 | 评价 |
|------|---------------------|------|
| PM | 3（S2, S5.4, S8.3） | 多触点 |
| 文档工程师 | 2（S5, S8.2） | 双触点 |
| 后端开发者 | 2（S5, S6） | 双触点 |
| **Harness 专家** | **1（S2）** | **孤立** |

**PM 和 Harness 的互补模型**：
- PM 发现"tasks.md 标记完成"→ Harness 验证"实际实现是否正确"
- PM 发现"build+test 绿"→ Harness 验证"202 返回后 run 是否被 context 取消（测试未覆盖）"
- PM 在 8.3 确认"tech-debt.md 存在"→ Harness 评估"tech debt 列表是否完整"

**PM 管"有没有"，Harness 管"对不对"。**

### 3.3 PM 检查中缺失的维度

当前 PM 只做：
1. build/test 二元检查（绿/红）
2. 产出物存在性检查（文件在不在）
3. tasks.md 标记检查

缺失：
1. **流程合规性**：spec 是否经过 4 角色审查？tasks.md 是否 Speckit 生成？
2. **跨阶段一致性**：spec 中 Harness 建议是否被落实到代码？
3. **内核集成质量**：Phase 1 的 5 个 P0 在 build+test 都绿时存在
4. **Tech debt 趋势**：是在增长还是收敛？

### 3.4 PM 提出的 3 个检查点

| 检查点 | 时机 | 内容 | 耗时 |
|--------|------|------|------|
| A: S2.5 审查完整性 | S2→S3 之间 | 4 份审查是否全部返回？有无 P0 级遗漏？ | 3-5 min |
| B: S5.5 集成抽检 | S5.4 之后 | 关键集成点代码是否正确？运行时行为有无缺陷？ | 5-10 min |
| C: S8.4 Phase 回顾 | S8.3 之后 | 流程偏差 + tech debt 完整性 + 改进建议 | 15-20 min |

### 3.5 职责边界

| | PM | Harness 专家 |
|--|---|---|
| 关注点 | 交付状态 | 执行质量 |
| 检查工具 | build/test 结果 + 文件存在性 | 代码逻辑 + 事件契约 + tech debt 趋势 |
| 阻塞权 | S8.3 PASS/FAIL | 内核健康度红色阻塞 |
| 类比 | SLE 中的 GateService | SLE 中的 ReviewService |

**冲突裁决规则**：
- 技术正确性问题 → Harness 专家优先
- 流程合规性问题 → PM 优先
- 优先级争议 → 总负责人裁决

## 4. Roadmap 规划师视角

### 4.1 工作流成熟度评估

| CMMI 级别 | 描述 | 当前状态 |
|-----------|------|---------|
| Level 1（初始） | 流程存在但被频繁绕过 | Phase 0-1 的状态 |
| **Level 2（已管理）** | 流程文档化 + 硬阻塞门 | **当前（Phase 2 后）** |
| Level 3（已定义） | 量化性能数据 + 主动改进循环 | **目标** |

达到 Level 3 的关键缺失：
1. **量化流程性能基线**：无每阶段耗时、返工率、教训复发率指标
2. **主动流程改进循环**：改进全部是被动的（问题暴露后修）
3. **跨 Phase 知识连续性**：教训跨 Phase 传递依赖 memory 和文档更新，但两者都曾被遗漏

### 4.2 反馈回路缺失分析

| 反馈类型 | 速度 | 范围 | 当前状态 |
|---------|------|------|---------|
| 快速局部 | 分钟级 | 单阶段内 | 部分（PM 5.4） |
| 慢速局部 | 小时级 | 阶段间 | 存在（S6 Review-Fix） |
| **慢速全局** | **Phase 级** | **跨 Phase** | **严重缺失** |
| **快速全局** | **分钟级** | **全流程** | **完全缺失** |

**Harness 专家的 Phase 回顾属于"慢速全局反馈"——当前最缺失的类型。**

还需要补充：
- 快速局部反馈扩展：S5 实施期间 Harness 对跨层集成 batch 做即时审查
- 前馈控制：S0 评估"集成风险等级"，据此调整流程参数

### 4.3 跨 Phase 知识传递的失效

| 机制 | 状态 | 问题 |
|------|------|------|
| memory 更新（S8.2） | 存在但被遗漏 | Phase 2.5 发现 memory 未更新 |
| workflow-detailed.md 更新 | 有效但被动 | 问题驱动，无法预防 |
| tech-debt 跨 Phase 追踪 | **缺失** | "本 Phase 修复"的 debt 无人检查 |
| 设计决策跨 Phase 一致性验证 | **缺失** | ADR-004 的 drift 无人检查 |
| 教训制度化转化 | **缺失** | 教训记录了但未转化为自动检查 |

**Harness 专家可以作为"跨 Phase 连续性守护者"**——在 S0 审查上一 Phase 的 tech-debt.md、phase-report.md、decisions.md 延迟项。

### 4.4 长期演进路线

| 阶段 | 时间 | 改进 | 验证指标 |
|------|------|------|---------|
| 近期 | Phase 3 | S0 连续性检查 + S5 集成检查点 + S8 Phase 回顾 | "用户提醒"次数=0 |
| 中期 | Phase 4-5 | 流程性能基线 + 入口/出口自动化 + tech-debt-registry 自动汇总 | 教训复发率下降 |
| 远期 | Phase 6+ | 流程参数可配置（简单 Phase 简化，复杂 Phase 加固）+ 教训自动化应用 | 流程偏差趋势下降 |

**Harness 专家终态角色**：工作流 SRE + 流程性能工程师 + 跨 Phase 连续性守护者

## 5. Harness 专家自我分析

### 5.1 缺位诊断

逐条分析历史教训：

| 教训 | 是否在我的职责范围？ | 我本应在哪个阶段发现？ |
|------|-------------------|---------------------|
| Phase 0 跳过 Harness 审查 | **直接相关** — 我被跳过 | S0 角色完整性检查 |
| Phase 1 跳过 Harness 审查 → 5 个 P0 | **直接相关** — 我又被跳过 | S2 审查 |
| QA 环境未准备 | **间接相关** | S2 应检查 spec 的 E2E 可测试性需求 |
| 文档+DevOps 缺失 | **间接相关** | S2 应检查 spec 的"执行完整性" |
| Speckit 被绕过 | **直接相关** | S4 应检查流水线合规性 |
| decisions.md 缺失 | 与我相关 | S3 — 我的建议采纳/拒绝无法追溯 |

### 5.2 SLE 自循环与工作流的类比

| SLE 管道 | 工作流对应 | 当前状态 | 缺失 |
|----------|-----------|---------|------|
| Execute | S5 实施 | 有 | — |
| Gate | PM 5.4 (build+test) | 部分 | S3/4/6/7/8 无机械检查 |
| Review | S6 (6 视角) | 有 | spec/tasks 后无 review |
| Triage | S6 P0/P1/P2 | 有 | — |
| Fix Loop | S6 最多 3 轮 | 有 | — |
| **反馈到下一循环** | **几乎为零** | **缺失** | **Phase 报告不回流** |

**关键洞察**：工作流有执行和观测，但**反馈和调整环节为零**。Harness 专家应该是工作流的"AfterGate hook"。

### 5.3 逐阶段参与方案

| 阶段 | 当前 | 建议 | 具体做什么 | 参与度 |

| S2 | **审查 spec** | 保持+结构化 | 建议 + 集成风险评估 + 可执行性评估 | **重度** |
| S3 | 无 | 裁决确认 | 被拒绝的 P0 建议是否有合理理由 | 轻度 |
| S4 | 无 | tasks 审查 | 内核集成任务是否存在 + Speckit 合规 | 中度 |
| S5 | 无 | 按需介入 | 涉及内核组件的 batch 快速审查 | 轻度 |
| S6 | 无 | 增加第 7 视角 | SLE 集成视角：边界/契约/一致性 | 中度 |
| S7 | 无 | QA 范围审查 | 管道端到端行为是否被覆盖 | 轻度 |
| S8 | 无 | Phase 回顾 | 7 维度评分 + 必须修复项 | **重度** |

### 5.5 防止自身被跳过的硬阻塞门

| 门 | 位置 | 条件 |
