## 必须修改（P0）

### 1. S8.3 双确认改为并行 + 分离回退路径

**问题**: 串行增加延迟；项目 FAIL 后重走产品确认不合理。
**方案**: 8.3-A 和 8.3-B 并行执行。单方 FAIL 只回到对应修复路径。双 PASS → 合并。
**来源**: 架构师 建议 1 + Harness 专家 建议 10

### 2. S5.4 验证方式修正

**问题**: feature branch 上 CI 不触发（`.github/workflows/ci.yml` 只在 push main/develop 或 PR 时触发）。
**方案**: 要求开发者 Agent 完成任务后自行运行 `go build + go test` 并在返回结果中报告 PASS/FAIL。项目经理读取开发者返回结果，不亲自运行。
**来源**: 项目经理 建议 1

### 3. S4 时序串行化

**问题**: 产品经理/项目经理并行产出（步骤 6）可能读到 Harness 追加前的 tasks.md。
**方案**: 明确步骤 5（Harness 审查）完成后才派发步骤 6。
**来源**: Harness 专家 建议 P0-1 + 项目经理 建议 2

### 4. 替换 80% AC 通过率为分级标准

**问题**: 5 条 AC 中 1 条核心功能 FAIL 但仍达 80%。SKIP 滥用风险。
**方案**: P1 AC（核心功能）= 100% PASS；P2 AC（增强）= 允许 SKIP 附理由；P3 AC（基础设施）= 允许 SKIP。
**来源**: 产品经理 建议 P0-1

### 5. S0 明确 product-context.md 输入来源

**问题**: Agent 被派发时如果缺少关键输入，persona 和成功标准是凭空编造的。
**方案**: S0 步骤 4 的 prompt 必须注入：Phase 目标描述 + PRD v2 + 上一 Phase product-review-report.md（如存在）。
**来源**: 产品经理 建议 P0-2
